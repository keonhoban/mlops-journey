## Airflow DAG CI: 파싱 오류를 운영 전에 잡는 구조

---

## 🧠 시나리오 설명

> "DAG 파일에 오타가 하나 들어갔습니다.
>
> Airflow Scheduler가 파싱 오류를 감지하고 해당 DAG을 비활성화합니다.
>
> 문제를 발견한 것은 배포 후 30분이 지난 시점입니다."
>
> "이 프로젝트에서는 이 상황을 방지하기 위해
>
> **DAG CI를 구성했습니다.**
>
> push 시점에 파싱 오류를 잡고, 운영 환경까지 가지 않도록 합니다."

---

## 🎯 핵심 요약

- **DAG CI 목적**: 파싱 오류 / import 오류를 배포 전에 감지
- **test_dag_integrity.py**: DagBag import, task 수 검증, SLA 설정 검증
- **GitHub Actions**: push/PR 시 자동 실행, Python 환경에서 테스트
- **로컬 실행**: `pytest tests/test_dag_integrity.py` 로 동일하게 검증 가능
- **CI 실패 시**: 오류 로그에서 파일명과 원인 즉시 확인 가능

---

## 1️⃣ 왜 DAG CI가 필요한가

Airflow DAG 파일은 Python 코드입니다.

다음과 같은 오류가 흔히 발생합니다.

```
import 오류 (없는 모듈 참조)
SyntaxError (오타, 들여쓰기 오류)
DAG 정의 오류 (dag_id 중복, schedule 형식 오류)
Task 의존성 오류 (순환 참조)
```

이 오류들은 Airflow Scheduler가 실행하기 전까지 발견되지 않습니다.

```
DAG 코드 수정
      ↓
Git push
      ↓
ArgoCD sync → Airflow에 반영
      ↓
Scheduler가 DAG 파일 파싱
      ↓
파싱 오류 발견 → DAG 비활성화
```

이 흐름에서 문제를 발견하는 것은 **배포 이후**입니다.

**파싱 오류 운영 중 발견 비용:**

```
파이프라인 중단
Scheduler 에러 로그 확인
코드 수정 → 재배포
최소 수십 분 소요
```

DAG CI는 이 비용을 push 시점으로 앞당깁니다.

```
코드 수정
      ↓
Git push
      ↓
GitHub Actions → test_dag_integrity.py 실행
      ↓
파싱 오류 즉시 감지 → PR 차단
```

## 2️⃣ test_dag_integrity.py 구조

관련 파일:

```
tests/test_dag_integrity.py
```

이 테스트 파일은 다음을 검증합니다.

```
DagBag import 검증
예상 DAG 존재 검증
Task 수 검증
e2e_full 전용 검증 (핵심 task, max_active_runs, catchup, SLA)
```

### (1) DagBag import 검증

```python
import pytest
from airflow.models import DagBag

DAG_FOLDER = "dags"

@pytest.fixture(scope="module")
def dagbag():
    return DagBag(dag_folder=DAG_FOLDER, include_examples=False)


def test_no_import_errors(dagbag):
    """DAG 파일 import 오류가 없어야 한다."""
    assert dagbag.import_errors == {}, (
        f"DAG import errors: {dagbag.import_errors}"
    )
```

`DagBag`은 Airflow가 DAG 파일을 로드하는 방식과 동일합니다.

`import_errors`가 비어 있지 않으면 테스트가 실패합니다.

오류 메시지에서 파일명과 원인을 바로 확인할 수 있습니다.

```
AssertionError: DAG import errors: {
  'dags/e2e_full.py': 'No module named mlops_lib.core.ids'
}
```

### (2) 예상 DAG 존재 검증

```python
EXPECTED_DAG_IDS = [
    "e2e_full",
    "dp_feature_pipeline",
    "feast_materialize_dev",
    "feast_full_refresh_dev_manual",
    "rollback_manual",
]


def test_expected_dag_ids_exist(dagbag):
    """예상된 모든 DAG이 정상 로드되어야 한다."""
    for dag_id in EXPECTED_DAG_IDS:
        assert dag_id in dagbag.dags, (
            f"Expected DAG '{dag_id}' not found in DagBag"
        )
```

이 검증으로 DAG 파일 삭제나 dag_id 변경이 배포 전에 감지됩니다.

### (3) Task 수 검증

```python
def test_dag_has_tasks(dagbag):
    """각 DAG에 최소 1개 이상의 Task가 있어야 한다."""
    for dag_id, dag in dagbag.dags.items():
        assert len(dag.tasks) > 0, (
            f"DAG '{dag_id}' has no tasks"
        )
```

### (4) e2e_full 핵심 task 검증

e2e_full DAG은 플랫폼의 핵심 파이프라인이므로

핵심 task가 반드시 존재해야 합니다.

```python
E2E_FULL_KEY_TASK_IDS = [
    "train_and_evaluate",
    "check_result",
    "observe_post_deploy_metrics",
    "rollback_minimal",
]


def test_e2e_full_key_task_ids(dagbag):
    """e2e_full DAG에 핵심 task가 모두 있어야 한다."""
    dag = dagbag.dags["e2e_full"]
    task_ids = {t.task_id for t in dag.tasks}
    for tid in E2E_FULL_KEY_TASK_IDS:
        assert tid in task_ids, (
            f"e2e_full: required task '{tid}' not found"
        )
```

`rollback_minimal` task가 없으면

배포 실패 시 자동 롤백이 동작하지 않습니다.

### (5) e2e_full 운영 설정 검증

```python
def test_e2e_full_max_active_runs(dagbag):
    """e2e_full은 동시 실행이 1개로 제한되어야 한다."""
    dag = dagbag.dags["e2e_full"]
    assert dag.max_active_runs == 1, (
        f"e2e_full max_active_runs={dag.max_active_runs}, expected 1"
    )


def test_e2e_full_catchup_disabled(dagbag):
    """e2e_full은 catchup이 비활성화되어야 한다."""
    dag = dagbag.dags["e2e_full"]
    assert dag.catchup is False, "e2e_full: catchup must be False"
```

`max_active_runs=1`은 동시에 두 번의 E2E 파이프라인이 실행되는 것을 방지합니다.

`catchup=False`는 과거 스케줄을 소급 실행하지 않도록 합니다.

### (6) e2e_full SLA 설정 검증

```python
from datetime import timedelta


def test_e2e_full_sla_in_default_args(dagbag):
    """e2e_full default_args에 SLA(dagrun_timeout)가 설정되어야 한다."""
    dag = dagbag.dags["e2e_full"]
    timeout = dag.dagrun_timeout
    assert timeout is not None, "e2e_full: dagrun_timeout is not set"
    assert timeout > timedelta(minutes=30), (
        f"e2e_full: dagrun_timeout={timeout} is too short"
    )


def test_e2e_full_sla_miss_callback(dagbag):
    """e2e_full에 sla_miss_callback이 설정되어야 한다."""
    dag = dagbag.dags["e2e_full"]
    assert dag.sla_miss_callback is not None, (
        "e2e_full: sla_miss_callback is not set"
    )
```

SLA miss 시 `sla_miss_callback`이 호출되어 Slack 알림이 전송됩니다.

`dagrun_timeout` 초과는 파이프라인이 hang 상태에 빠졌음을 의미합니다.

## 3️⃣ GitHub Actions workflow 구조

관련 파일:

```
.github/workflows/ci.yml
```

전체 구조:

```yaml
name: DAG Integrity Test

on:
  push:
    branches:
      - main
      - dev
    paths:
      - 'dags/**'
      - 'tests/**'
  pull_request:
    branches:
      - main
      - dev
    paths:
      - 'dags/**'
      - 'tests/**'

jobs:
  dag-integrity:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.10'

      - name: Install dependencies
        run: |
          pip install apache-airflow==2.9.0 pytest

      - name: Run DAG integrity tests
        env:
          AIRFLOW__CORE__LOAD_EXAMPLES: 'False'
          AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: 'sqlite:////tmp/airflow_test.db'
        run: |
          airflow db migrate
          pytest tests/test_dag_integrity.py -v
```

### 주요 설계 포인트

**paths 필터:**

```yaml
paths:
  - 'dags/**'
  - 'tests/**'
```

DAG 파일 또는 테스트 파일이 변경될 때만 CI가 실행됩니다.

README 수정 등 불필요한 실행을 방지합니다.

**SQLite DB 사용:**

```
AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: 'sqlite:////tmp/airflow_test.db'
```

CI 환경에서 PostgreSQL 없이 SQLite로 Airflow DB를 구성합니다.

`airflow db migrate`로 스키마를 초기화한 후 테스트를 실행합니다.

**LOAD_EXAMPLES 비활성화:**

```
AIRFLOW__CORE__LOAD_EXAMPLES: 'False'
```

Airflow 기본 예시 DAG가 로드되지 않도록 합니다.

테스트 범위를 프로젝트 DAG만으로 한정합니다.

## 4️⃣ CI 실패 시나리오와 대응

### 시나리오 1: import 오류

```python
# dags/mlops_train.py
from mlops_lib.feature import build_features  # 존재하지 않는 모듈
```

CI 실패 메시지:

```
FAILED tests/test_dag_integrity.py::test_no_import_errors
AssertionError: DAG import errors: {
  'dags/mlops_train.py': "No module named 'mlops_lib.feature'"
}
```

대응:

```
1. 오류 메시지에서 파일명과 모듈명 확인
2. 해당 파일에서 import 경로 수정
3. 로컬에서 pytest 재실행 후 push
```

### 시나리오 2: SyntaxError

```python
# dags/mlops_deploy.py
def train_task()
    pass  # 콜론 누락
```

CI 실패 메시지:

```
FAILED tests/test_dag_integrity.py::test_no_import_errors
AssertionError: DAG import errors: {
  'dags/mlops_deploy.py': "invalid syntax (mlops_deploy.py, line 5)"
}
```

대응:

```
1. 오류 메시지에서 파일명과 라인 번호 확인
2. 해당 라인 수정
3. 로컬에서 pytest 재실행
```

### 시나리오 3: e2e_full SLA 미설정

```
FAILED tests/test_dag_integrity.py::test_e2e_full_sla_miss_callback
AssertionError: e2e_full: sla_miss_callback is not set
```

대응:

```python
# e2e_full.py DAG 정의에 sla_miss_callback 추가
from mlops_lib.alert import notify_sla_miss

with DAG(
    dag_id="e2e_full",
    sla_miss_callback=notify_sla_miss,
    dagrun_timeout=timedelta(hours=2),
    ...
) as dag:
    ...
```

### 시나리오 4: 예상 DAG 없음

```
FAILED tests/test_dag_integrity.py::test_expected_dag_ids_exist
AssertionError: Expected DAG 'rollback_manual' not found in DagBag
```

대응:

```
1. dags/rollback_manual.py 파일 존재 여부 확인
2. dag_id 문자열이 "rollback_manual"과 일치하는지 확인
3. import 오류 없는지 test_no_import_errors 먼저 확인
```

## 🧩 팁

- `DagBag`은 Airflow Scheduler가 DAG을 로드하는 방식과 동일합니다. CI에서 같은 방식으로 검증하면 운영과의 차이가 없습니다.
- `include_examples=False`를 반드시 설정해야 Airflow 기본 DAG이 테스트에 포함되지 않습니다.
- CI 환경에서 `AIRFLOW__CORE__DAGS_FOLDER`를 DAG_FOLDER와 일치시키면 더 정확한 검증이 가능합니다.
- SLA 검증 대상 목록(`SLA_REQUIRED_DAGS`)은 운영 DAG이 추가될 때 함께 관리해야 합니다.

## 🏁 정리

| 검증 항목 | 테스트 함수 | 감지 오류 |
| --- | --- | --- |
| import 오류 | `test_no_import_errors` | SyntaxError, ModuleNotFoundError 등 |
| DAG 존재 | `test_expected_dag_ids_exist` | DAG 삭제, dag_id 변경 |
| Task 수 | `test_dag_has_tasks` | 빈 DAG 정의 |
| 핵심 task | `test_e2e_full_key_task_ids` | rollback_minimal 누락 등 |
| 동시 실행 제한 | `test_e2e_full_max_active_runs` | max_active_runs 미설정 |
| catchup | `test_e2e_full_catchup_disabled` | 소급 실행 위험 |
| SLA timeout | `test_e2e_full_sla_in_default_args` | dagrun_timeout 미설정 |
| SLA callback | `test_e2e_full_sla_miss_callback` | sla_miss_callback 누락 |

> DAG CI는 파싱 오류를 운영 환경 배포 전에 잡습니다.
>
> `DagBag`으로 실제 로드 방식과 동일하게 검증하고,
>
> GitHub Actions로 push 시점에 자동 실행합니다.
>
> 로컬에서도 `pytest tests/test_dag_integrity.py -v` 하나로 동일하게 확인할 수 있습니다.
