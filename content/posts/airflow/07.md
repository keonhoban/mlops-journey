## 이 단계에서 해결하려는 문제

DAG 파일에 오타가 하나 들어가면 Airflow Scheduler가 파싱 오류를 감지하고 해당 DAG을 비활성화한다. 문제를 발견하는 것은 배포 후 수십 분이 지난 시점이다. 이 비용을 push 시점으로 앞당겨서 파싱 오류를 운영 전에 잡는 구조를 만든다.

---

### 🎯 핵심 요약

- **DAG CI 목적**: 파싱 오류 / import 오류를 배포 전에 감지
- **test_dag_integrity.py**: DagBag import, task 수 검증, SLA 설정 검증
- **GitHub Actions**: push/PR 시 자동 실행, Python 환경에서 테스트
- **로컬 실행**: `pytest tests/test_dag_integrity.py` 로 동일하게 검증 가능
- **CI 실패 시**: 오류 로그에서 파일명과 원인 즉시 확인 가능

---

## 1️⃣ 왜 DAG CI가 필요한가

Airflow DAG 파일은 Python 코드이므로 import 오류, SyntaxError, DAG 정의 오류, 순환 참조 등이 발생할 수 있다. 이 오류들은 Scheduler가 파싱하기 전까지 발견되지 않는다.

```
DAG CI 없이:  코드 수정 → push → ArgoCD sync → Scheduler 파싱 → 오류 발견 (배포 후)
DAG CI 적용:  코드 수정 → push → GitHub Actions → 오류 즉시 감지 → PR 차단 (배포 전)
```

## 2️⃣ test_dag_integrity.py 구조

### (1) DagBag import 검증

```python
import pytest
from airflow.models import DagBag

DAG_FOLDER = "dags"

@pytest.fixture(scope="module")
def dagbag():
    return DagBag(dag_folder=DAG_FOLDER, include_examples=False)

def test_no_import_errors(dagbag):
    """DAG 파일 import 오류가 없어야 한다."""
    assert dagbag.import_errors == {}, (
        f"DAG import errors: {dagbag.import_errors}"
    )
```

`DagBag`은 Airflow Scheduler가 DAG을 로드하는 방식과 동일하다.

### (2) 예상 DAG 존재 검증

```python
EXPECTED_DAG_IDS = [
    "e2e_full",
    "dp_feature_pipeline",
    "feast_materialize_dev",
    "feast_full_refresh_dev_manual",
    "rollback_manual",
]

def test_expected_dag_ids_exist(dagbag):
    """예상된 모든 DAG이 정상 로드되어야 한다."""
    for dag_id in EXPECTED_DAG_IDS:
        assert dag_id in dagbag.dags, (
            f"Expected DAG '{dag_id}' not found in DagBag"
        )
```

### (3) Task 수 검증

```python
def test_dag_has_tasks(dagbag):
    """각 DAG에 최소 1개 이상의 Task가 있어야 한다."""
    for dag_id, dag in dagbag.dags.items():
        assert len(dag.tasks) > 0, f"DAG '{dag_id}' has no tasks"
```

### (4) e2e_full 핵심 task 검증

```python
E2E_FULL_KEY_TASK_IDS = [
    "train_and_evaluate",
    "check_result",
    "observe_post_deploy_metrics",
    "rollback_minimal",
]

def test_e2e_full_key_task_ids(dagbag):
    dag = dagbag.dags["e2e_full"]
    task_ids = {t.task_id for t in dag.tasks}
    for tid in E2E_FULL_KEY_TASK_IDS:
        assert tid in task_ids, f"e2e_full: required task '{tid}' not found"
```

### (5) e2e_full 운영 설정 검증

```python
def test_e2e_full_max_active_runs(dagbag):
    dag = dagbag.dags["e2e_full"]
    assert dag.max_active_runs == 1

def test_e2e_full_catchup_disabled(dagbag):
    dag = dagbag.dags["e2e_full"]
    assert dag.catchup is False
```

### (6) e2e_full SLA 설정 검증

```python
from datetime import timedelta

def test_e2e_full_sla_in_default_args(dagbag):
    dag = dagbag.get_dag("e2e_full")
    sla = dag.default_args.get("sla")
    assert sla is not None
    assert isinstance(sla, timedelta)
    assert sla > timedelta(minutes=30)

def test_e2e_full_sla_miss_callback(dagbag):
    dag = dagbag.get_dag("e2e_full")
    assert dag.sla_miss_callback is not None
```

## 3️⃣ GitHub Actions workflow 구조

```yaml
# .github/workflows/ci.yml
name: DAG Integrity

on:
  push:
    branches: ["**"]
  pull_request:
    branches: ["**"]

jobs:
  test:
    name: DagBag Integrity Test
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: |
          pip install --upgrade pip
          if [ -f requirements.txt ]; then
            pip install -r requirements.txt
          else
            pip install "apache-airflow>=2.10.0,<3.0" pytest
          fi

      - name: Initialize Airflow DB
        run: |
          export AIRFLOW_HOME="${GITHUB_WORKSPACE}/.airflow"
          airflow db init
        env:
          AIRFLOW__CORE__DAGS_FOLDER: ${{ github.workspace }}/dags
          AIRFLOW__CORE__LOAD_EXAMPLES: "False"

      - name: Run DAG integrity tests
        run: pytest tests/test_dag_integrity.py -v --tb=short
        env:
          AIRFLOW_HOME: ${{ github.workspace }}/.airflow
          AIRFLOW__CORE__DAGS_FOLDER: ${{ github.workspace }}/dags
          AIRFLOW__CORE__LOAD_EXAMPLES: "False"
          MLFLOW_TRACKING_URI: "sqlite:///mlflow.db"
```

### 주요 설계 포인트

- **모든 브랜치 트리거**: `branches: ["**"]` — feature 브랜치에서도 즉시 피드백
- **AIRFLOW_HOME 분리**: CI 워크스페이스 내 격리된 Airflow 디렉토리 사용
- **LOAD_EXAMPLES 비활성화**: Airflow 기본 예시 DAG 제외
- **MLFLOW_TRACKING_URI 더미값**: DAG import 시 환경변수 참조 오류 방지

## 4️⃣ CI 실패 시나리오와 대응

### 시나리오 1: import 오류

```
FAILED test_no_import_errors
AssertionError: DAG import errors: {
  'dags/mlops_train.py': "No module named 'mlops_lib.feature'"
}
```

→ 파일명과 모듈명 확인 후 import 경로 수정

### 시나리오 2: SyntaxError

```
FAILED test_no_import_errors
AssertionError: DAG import errors: {
  'dags/mlops_deploy.py': "invalid syntax (mlops_deploy.py, line 5)"
}
```

→ 해당 라인 수정

### 시나리오 3: e2e_full SLA 미설정

```
FAILED test_e2e_full_sla_miss_callback
AssertionError: e2e_full: sla_miss_callback is not set
```

→ DAG 정의에 `sla_miss_callback` 추가

### 시나리오 4: 예상 DAG 없음

```
FAILED test_expected_dag_ids_exist
AssertionError: Expected DAG 'rollback_manual' not found in DagBag
```

→ 파일 존재 여부 확인, dag_id 일치 확인, import 오류 여부 확인

## 🧩 팁

- `DagBag`은 Airflow Scheduler가 DAG을 로드하는 방식과 동일하다. CI에서 같은 방식으로 검증하면 운영과의 차이가 없다.
- `include_examples=False`를 반드시 설정해야 Airflow 기본 DAG이 테스트에 포함되지 않는다.
- SLA 검증 대상 목록(`EXPECTED_DAG_IDS`)은 운영 DAG이 추가될 때 함께 관리해야 한다.

---

## 설계 판단 (Why This Way?)

DagBag 기반 테스트는 Scheduler와 동일한 파싱 메커니즘을 CI에서 재현하여 import 오류와 런타임 설정 오류를 배포 전에 잡는다. 핵심 파이프라인 e2e_full은 max_active_runs, catchup, rollback task 존재 여부까지 별도로 검증한다.
