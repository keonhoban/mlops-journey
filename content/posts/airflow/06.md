## Airflow KubernetesExecutor: Task 격리와 동적 리소스 할당

---

## 🧠 시나리오 설명

> "Airflow를 Kubernetes 위에 올리면 Executor가 세 가지입니다.
>
>
> LocalExecutor, CeleryExecutor, KubernetesExecutor.
>
> 이 중에서 MLOps 파이프라인에 가장 적합한 것을 선택하는 과정은
>
> **단순한 설치 옵션 선택이 아니라 운영 구조에 대한 결정**입니다."
>
> "이 프로젝트에서는 CeleryExecutor 대신 KubernetesExecutor를 선택했습니다.
>
> 이 글은 그 이유와 실제 구성 방법을 설명합니다."

---

### 🎯 핵심 요약

- **Executor 선택 기준**: Task 격리 / 동적 리소스 / Worker 관리 비용
- **KubernetesExecutor 선택 이유**
    - Task마다 독립 Pod → 환경 격리 보장
    - Broker(Redis/RabbitMQ) 불필요 → 운영 복잡도 감소
    - Idle Worker 없음 → 리소스 낭비 제거
    - Task 단위 리소스 정의 → ML 학습/서빙 작업에 적합
- **실제 구성 요소**
    - ServiceAccount + RBAC (Pod 생성 권한)
    - pod_template_file (Task Pod 표준 정의)
    - KubernetesPodOperator (커스텀 Task Pod)
- **운영 주의사항**
    - Cold start: Pod 스케줄링 지연 (~10~30s)
    - Image pull policy: IfNotPresent 기본 권장
    - 로그 지속성: NFS 또는 Remote Logging 필수

## 1️⃣ Executor 종류 비교

```
Executor 비교
┌──────────────────┬──────────────────────────────────────────────────────────────────────┐
│ LocalExecutor    │ Scheduler가 Task를 직접 실행. 단일 머신. 병렬 제한.                  │
│                  │ 소규모 환경에서만 적합.                                              │
├──────────────────┼──────────────────────────────────────────────────────────────────────┤
│ CeleryExecutor   │ Broker(Redis/RabbitMQ) + 별도 Worker 프로세스.                       │
│                  │ Worker가 항상 실행 중 → 유휴 리소스 발생.                            │
│                  │ Broker 관리 비용 추가.                                               │
├──────────────────┼──────────────────────────────────────────────────────────────────────┤
│ KubernetesExecutor│ Task마다 Pod를 생성하고 완료 후 삭제.                               │
│                  │ Broker 불필요. Worker 관리 불필요.                                   │
│                  │ Task 격리, 동적 리소스 정의 가능.                                    │
└──────────────────┴──────────────────────────────────────────────────────────────────────┘
```

---

## 2️⃣ KubernetesExecutor를 선택한 이유

### (1) Task 격리

CeleryExecutor의 Worker는 프로세스 단위로 격리됩니다.

즉, 같은 Worker Pod에서 다른 Task들이 동일한 Python 환경을 공유합니다.

```
CeleryExecutor
Worker Pod → Task A, Task B, Task C (같은 환경 공유)
```

KubernetesExecutor는 Task마다 Pod가 분리됩니다.

```
KubernetesExecutor
Task A → Pod-A (독립 실행, 완료 후 삭제)
Task B → Pod-B (독립 실행, 완료 후 삭제)
Task C → Pod-C (독립 실행, 완료 후 삭제)
```

ML 파이프라인에서는 Task마다 패키지 의존성이 다를 수 있습니다.

예를 들어 feature 생성 Task와 model 학습 Task는 서로 다른 라이브러리가 필요할 수 있습니다.

KubernetesExecutor는 이 문제를 구조적으로 해결합니다.

### (2) Broker 불필요

CeleryExecutor는 반드시 Broker가 필요합니다.

```
CeleryExecutor 구조
Scheduler → Redis/RabbitMQ(Broker) → Worker
```

Broker는 별도 서비스이므로 다음이 추가됩니다.

- Broker 설치 및 운영
- Broker 장애 대응
- Broker 모니터링

KubernetesExecutor는 Broker 없이 Kubernetes API를 직접 사용합니다.

```
KubernetesExecutor 구조
Scheduler → Kubernetes API → Task Pod
```

운영 컴포넌트가 줄어든다는 것은 장애 발생 지점이 줄어드는 것과 같습니다.

### (3) Idle Worker 없음

CeleryExecutor는 Worker가 항상 실행 중입니다.

DAG가 실행되지 않는 시간에도 Worker Pod는 리소스를 점유합니다.

KubernetesExecutor는 Task가 실행될 때만 Pod가 생성됩니다.

이 구조는 리소스 비용 측면에서 유리합니다.

### (4) Task 단위 리소스 정의

ML Task는 리소스 요구사항이 제각각입니다.

```
feature 생성 Task: CPU 낮음, Memory 보통
model 학습 Task:   CPU 높음, Memory 많음
drift 검사 Task:   CPU 낮음, Memory 낮음
```

KubernetesExecutor는 Task마다 리소스를 다르게 정의할 수 있습니다.

```python
from airflow.providers.cncf.kubernetes.operators.pod import KubernetesPodOperator
from kubernetes.client import models as k8s

train_task = KubernetesPodOperator(
    task_id="model_training",
    name="model-training-pod",
    namespace="airflow-dev",
    image="my-ml-image:latest",
    resources=k8s.V1ResourceRequirements(
        requests={"cpu": "2", "memory": "4Gi"},
        limits={"cpu": "4", "memory": "8Gi"},
    ),
    ...
)
```

## 3️⃣ 실제 RBAC + ServiceAccount 구성

KubernetesExecutor가 Task Pod를 생성하려면 Kubernetes RBAC 권한이 필요합니다.

### (1) ServiceAccount

```yaml
# envs/dev/airflow/serviceaccount.yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: airflow-dev
  namespace: airflow-dev
```

### (2) Role (Pod 생성/삭제/조회 권한)

```yaml
# envs/dev/airflow/role.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: airflow-dev-role
  namespace: airflow-dev
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch", "create", "delete", "patch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get", "list"]
  - apiGroups: [""]
    resources: ["pods/exec"]
    verbs: ["create"]
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["get", "list"]
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list"]
```

### (3) RoleBinding

```yaml
# envs/dev/airflow/rolebinding.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: airflow-dev-rolebinding
  namespace: airflow-dev
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: airflow-dev-role
subjects:
  - kind: ServiceAccount
    name: airflow-dev
    namespace: airflow-dev
```

> Airflow Scheduler가 실행 중인 Namespace와 Task Pod가 생성되는 Namespace가 같다면 Role이 충분합니다.
>
> 다른 Namespace에 Task Pod를 생성해야 한다면 ClusterRole과 ClusterRoleBinding이 필요합니다.

## 4️⃣ pod_template_file: Task Pod 표준 정의

KubernetesExecutor는 Task Pod의 기본 템플릿을 `pod_template_file`로 정의합니다.

이 파일이 없으면 기본값으로 동작하지만,

**운영 환경에서는 반드시 명시적으로 정의**하는 것이 좋습니다.

```yaml
# pod_template.yaml
apiVersion: v1
kind: Pod
metadata:
  name: base-worker
  namespace: airflow-dev
spec:
  serviceAccountName: airflow-dev
  restartPolicy: Never
  containers:
    - name: base
      image: apache/airflow:2.9.0
      imagePullPolicy: IfNotPresent
      resources:
        requests:
          cpu: "500m"
          memory: "512Mi"
        limits:
          cpu: "1"
          memory: "1Gi"
      volumeMounts:
        - name: airflow-logs
          mountPath: /opt/airflow/logs
  volumes:
    - name: airflow-logs
      persistentVolumeClaim:
        claimName: airflow-logs-pvc-dev
```

이 템플릿은 다음을 표준화합니다.

- ServiceAccount: RBAC 권한 자동 부여
- ImagePullPolicy: 불필요한 이미지 재다운로드 방지
- Resources: 기본 리소스 보장
- Logs PVC: Task 로그 NFS 영구 저장

Airflow Helm values에서 다음과 같이 참조합니다.

```yaml
# charts/airflow/values-dev.yaml
config:
  kubernetes:
    pod_template_file: /opt/airflow/pod_template.yaml
    worker_container_name: base
    namespace: airflow-dev
```

## 5️⃣ KubernetesPodOperator (KPO) 실전 예시

Task마다 다른 이미지, 다른 리소스를 사용해야 할 때 KPO를 활용합니다.

```python
from airflow import DAG
from airflow.providers.cncf.kubernetes.operators.pod import KubernetesPodOperator
from kubernetes.client import models as k8s
from datetime import datetime

with DAG(
    dag_id="kpo_example_dev",
    start_date=datetime(2026, 1, 1),
    schedule=None,
    catchup=False,
    tags=["kpo", "dev"],
) as dag:

    feature_task = KubernetesPodOperator(
        task_id="build_features",
        name="feature-builder",
        namespace="airflow-dev",
        image="my-feature-image:1.0.0",
        image_pull_policy="IfNotPresent",
        service_account_name="airflow-dev",
        resources=k8s.V1ResourceRequirements(
            requests={"cpu": "500m", "memory": "512Mi"},
            limits={"cpu": "1", "memory": "1Gi"},
        ),
        env_vars={
            "S3_BUCKET": "my-feature-bucket",
            "ENV": "dev",
        },
        is_delete_operator_pod=True,      # 완료 후 Pod 삭제
        get_logs=True,                    # Task 로그 Airflow UI에 표시
        do_xcom_push=False,
    )

    train_task = KubernetesPodOperator(
        task_id="train_model",
        name="model-trainer",
        namespace="airflow-dev",
        image="my-train-image:1.0.0",
        image_pull_policy="IfNotPresent",
        service_account_name="airflow-dev",
        resources=k8s.V1ResourceRequirements(
            # 학습 Task는 리소스를 더 많이 할당
            requests={"cpu": "2", "memory": "4Gi"},
            limits={"cpu": "4", "memory": "8Gi"},
        ),
        env_vars={
            "MLFLOW_TRACKING_URI": "http://mlflow-dev-service.mlflow-dev.svc.cluster.local:5000",
        },
        is_delete_operator_pod=True,
        get_logs=True,
    )

    feature_task >> train_task
```

> `is_delete_operator_pod=True`는 Task 완료 후 Pod를 정리합니다.
>
> 운영 환경에서는 반드시 설정해서 Pod 누적을 방지해야 합니다.

## 6️⃣ 운영 주의사항

### (1) Cold Start 지연

KubernetesExecutor는 Task 실행 시 Pod를 새로 생성합니다.

```
Task 트리거 → Pod 생성 요청 → 스케줄링 → 이미지 Pull → 컨테이너 시작 → Task 실행
```

이 과정은 **10~30초 이상** 소요될 수 있습니다.

특히 이미지 크기가 크거나 Node에 이미지가 없으면 더 길어집니다.

**완화 방법**

```yaml
imagePullPolicy: IfNotPresent  # 이미지가 이미 있으면 Pull 생략
```

Node에 이미지를 미리 캐싱하거나,

경량 이미지를 사용하면 Cold start를 줄일 수 있습니다.

### (2) 로그 지속성

Task Pod는 완료 후 삭제됩니다.

Pod가 삭제되면 Pod 내부의 로그도 함께 사라집니다.

**반드시 다음 중 하나를 선택해야 합니다.**

```
옵션 A: NFS PVC 마운트 → Task 로그를 공유 스토리지에 기록
옵션 B: Remote Logging → S3, GCS 등 원격 저장소에 직접 기록 (권장)
```

이 프로젝트에서는 초기 구성으로 NFS PVC 마운트를 사용합니다.

```yaml
# pod_template.yaml
volumeMounts:
  - name: airflow-logs
    mountPath: /opt/airflow/logs
volumes:
  - name: airflow-logs
    persistentVolumeClaim:
      claimName: airflow-logs-pvc-dev
```

### (3) RBAC 권한 범위

Task Pod 생성에 필요한 최소 권한만 부여해야 합니다.

과도한 권한(ClusterAdmin 등)은 보안 위험입니다.

```
권장: Role (Namespace 범위) + RoleBinding
비권장: ClusterRole + ClusterRoleBinding (전체 클러스터 범위)
```

### (4) 동시 실행 Task 수 제한

KubernetesExecutor는 동시에 많은 Pod를 생성할 수 있습니다.

클러스터 리소스가 부족하면 Pod가 Pending 상태에 걸립니다.

Airflow 설정에서 병렬도를 제한합니다.

```yaml
config:
  core:
    parallelism: 16           # 전체 동시 Task 수
    max_active_tasks_per_dag: 8  # DAG당 동시 Task 수
```

## 7️⃣ 검증 체크리스트

```bash
# ServiceAccount 생성 확인
kubectl -n airflow-dev get serviceaccount airflow-dev

# Role/RoleBinding 확인
kubectl -n airflow-dev get role,rolebinding | grep airflow

# Task Pod 생성 테스트 (DAG 실행 후)
kubectl -n airflow-dev get pods --watch

# Task Pod 로그 확인
kubectl -n airflow-dev logs <task-pod-name> -c base

# NFS 로그 PVC Bound 확인
kubectl -n airflow-dev get pvc airflow-logs-pvc-dev
```

## 🧩 트러블슈팅

### (1) Task Pod가 Pending 상태에서 멈춤

```bash
kubectl -n airflow-dev describe pod <task-pod-name>
# Events 항목에서 원인 확인
# Insufficient cpu/memory → 리소스 부족
# No matching node → NodeSelector/Affinity 설정 확인
```

### (2) `Forbidden: pods is forbidden` 에러

ServiceAccount에 Pod 생성 권한이 없는 경우입니다.

```bash
kubectl -n airflow-dev auth can-i create pods --as=system:serviceaccount:airflow-dev:airflow-dev
# yes → 권한 있음
# no  → Role/RoleBinding 확인 필요
```

### (3) Task 로그가 Airflow UI에 표시되지 않음

Remote Logging 미설정 상태에서 Pod가 삭제된 경우입니다.

```bash
# NFS 마운트 확인
kubectl -n airflow-dev exec <scheduler-pod> -- ls /opt/airflow/logs
```

### (4) 이미지 Pull이 느린 경우

```yaml
imagePullPolicy: IfNotPresent  # Always → IfNotPresent 변경
```

또는 Private Registry에서 ImagePullSecret을 설정합니다.

```yaml
spec:
  imagePullSecrets:
    - name: registry-secret
```

## 🔧 MLOps 실전 연결

KubernetesExecutor는 ML 파이프라인 운영에서 다음을 가능하게 합니다.

- **환경 격리**: feature 생성, 모델 학습, drift 검사 Task가 서로 다른 이미지/환경에서 실행
- **리소스 효율**: 학습 Task는 고사양, 검증 Task는 저사양으로 개별 정의
- **Broker 제거**: CeleryExecutor의 Redis 브로커 운영 부담 없이 동일한 병렬 실행 가능
- **K8s 네이티브**: PVC, Secret, ConfigMap을 Task Pod에 직접 마운트 가능

---

## 🏁 정리

| 항목 | LocalExecutor | CeleryExecutor | KubernetesExecutor |
| --- | --- | --- | --- |
| Broker 필요 | 없음 | Redis/RabbitMQ | 없음 |
| Worker 상시 실행 | - | 있음 | 없음 |
| Task 격리 | 프로세스 | 프로세스 | Pod |
| 리소스 유연성 | 낮음 | 낮음 | 높음 |
| K8s 네이티브 | 낮음 | 낮음 | 높음 |
| 운영 복잡도 | 낮음 | 중간 | 중간 |

> MLOps 파이프라인에서 Task 격리와 동적 리소스 할당이 필요하다면
>
> KubernetesExecutor가 가장 적합한 선택입니다.
>
> Broker 없이 Kubernetes API만으로 동작하므로 운영 컴포넌트도 줄어듭니다.
