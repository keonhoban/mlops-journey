## 이 단계에서 해결하려는 문제

Airflow DAG 내부에서 `mlflow.sklearn.log_model()`을 호출하고 Model Registry에 연동하는 과정에서, 볼륨 마운트 누락, API 미활성화, CLI 설정 오류 등 환경 구성 문제가 빈번하게 발생합니다. 이 문제들은 공식 문서만으로는 원인을 파악하기 어렵습니다.

---

## 에러 1. PermissionError: [Errno 13] Permission denied: '/mlflow'

### 원인 분석

| 증상 | 원인 |
| --- | --- |
| `/mlflow` 경로 접근 실패 | Airflow 컨테이너에 mount되지 않음 |
| `mlruns` 쓰기 실패 | 공유 볼륨 누락 → 쓰기 권한 없음 |
| 실험 생성 실패 | artifact 저장소 지정 실패 |

### 해결 방법

1. **Airflow 컨테이너에 공유 볼륨 추가**

```yaml
volumes:
  - ./mlflow:/mlflow
  - ./mlruns:/mlflow/mlruns
```

2. **초기 실험 수동 생성 스크립트 실행**

```python
# init_experiments.py
from mlflow.tracking import MlflowClient

mlflow.set_tracking_uri("http://localhost:5000")
client = MlflowClient()

if not client.get_experiment_by_name("Iris_Model_Registry_Go"):
    client.create_experiment(
        name="Iris_Model_Registry_Go",
        artifact_location="file:///mlflow/mlruns/Iris_Model_Registry_Go"
    )
```

```bash
docker exec -it airflow_mlflow_1 python3 /mlflow/init_experiments.py
```

3. **Airflow DAG 내에서 동일한 이름 지정**

```python
mlflow.set_experiment("Iris_Model_Registry_Go")
```

## 에러 2. 404 Not Found on /api/2.0/mlflow/logged-models

### 원인 분석

| 핵심 문제 | 상세 설명 |
| --- | --- |
| Model Registry API 미탑재 | `mlflow ui` 또는 `mlflow server`의 옵션 누락 |
| `/logged-models` 경로 없음 | 기본 실행 시에는 Registry 관련 API가 비활성화 |

### 해결 방법

| 실행 방식 | Registry API |
| --- | --- |
| `mlflow ui` | 없음 |
| `mlflow server` | 일부 없음 |
| `mlflow server --serve-artifacts` | 모두 활성화 |
| `mlflow[extras]` 패키지 | 커스텀 이미지 사용 필요 |

Dockerfile 예시 (커스텀 구성):

```
FROM python:3.10-slim
RUN pip install mlflow[extras]
CMD ["mlflow", "server", "--serve-artifacts", "--host", "0.0.0.0", "--port", "5000"]
```

## 에러 3. Airflow CLI 오류: the following arguments are required: GROUP_OR_COMMAND

### 원인 분석

| 증상 | 원인 |
| --- | --- |
| Airflow 컨테이너 실행 직후 종료 | 명령어 없이 airflow 실행됨 |
| `GROUP_OR_COMMAND` 요구 | `CMD`, `ENTRYPOINT` 누락 |

### 해결 방법

- **개발/테스트 용도**: `standalone` 모드 사용

```yaml
command: standalone
```

- **실무 구성**: `webserver`, `scheduler` 분리 실행

```yaml
command: webserver
# 또는
command: scheduler
```

## 공유 볼륨 구조 요약

| Host 디렉토리 | MLflow 컨테이너 | Airflow 컨테이너 |
| --- | --- | --- |
| `./mlflow` | `/mlflow` | `/mlflow` |
| `./mlruns` | `/mlflow/mlruns` | `/mlflow/mlruns` |

공통 저장소 구조가 필수입니다 — 실험 로깅, 모델 추론, 등록 경로가 통일되어야 합니다.

---

## 설계 판단 (Why This Way?)

- **Docker 볼륨 공유 설계 원칙**: Airflow와 MLflow 컨테이너가 동일한 파일 시스템 경로를 사용해야 합니다. 한쪽에서 `/mlflow/mlruns`에 기록한 아티팩트를 다른 쪽에서도 동일 경로로 접근할 수 있어야 모델 로딩이 가능합니다.
- **`--serve-artifacts` 옵션의 필요성**: 기본 `mlflow server`는 Tracking API만 제공합니다. Registry API와 아티팩트 프록시를 활성화하려면 `--serve-artifacts`가 필수입니다. 이 옵션 없이는 모델 등록/다운로드가 실패합니다.
- **standalone vs webserver/scheduler 분리**: 개발 환경에서는 `standalone`이 편리하지만, 프로덕션에서는 webserver와 scheduler를 분리하여 각각 스케일링하고 장애를 격리합니다.
- **실험 디렉토리 사전 생성**: `mlflow.set_experiment()`은 실험이 존재하지 않으면 자동 생성하지만, artifact_location이 올바르게 설정되지 않을 수 있습니다. 초기 실험을 명시적으로 생성하면 경로 문제를 예방합니다.

---

## 면접 포인트

- **Q: MLflow에서 `--serve-artifacts` 옵션의 역할은?**
  → MLflow 서버가 아티팩트 저장소에 대한 프록시 역할을 합니다. 클라이언트가 직접 S3/NFS에 접근하지 않아도 서버를 통해 아티팩트를 업로드/다운로드할 수 있으며, Registry API도 활성화됩니다.

- **Q: 컨테이너 간 볼륨 공유 시 주의할 점은?**
  → 마운트 경로가 컨테이너 간 동일해야 합니다. 경로가 다르면 한쪽에서 기록한 파일을 다른 쪽에서 찾을 수 없습니다. UID/GID 불일치로 인한 권한 오류도 흔한 문제이므로, 컨테이너 실행 사용자의 권한을 사전에 확인해야 합니다.
