## 🛠 Helm 구성

### values.yaml (핵심 설정)

```yaml
# 실행 엔진 설정
executor: KubernetesExecutor

# 이미지 설정
images:
  airflow:
    repository: hoizz/airflow-custom
    tag: v5-mlflow-20240712
    pullPolicy: IfNotPresent

# Fernet Key 고정 (helm delete & install 이후에도 Variables 등 데이터 유지)
fernetKeySecretName: my-fernet-secret

# DAG GitSync 설정 (GitOps 기반)
dags:
  gitSync:
    enabled: true
    repo: git@github.com:keonhoban/airflow-dags.git
    branch: main
    subPath: dags
    depth: 1
    wait: 10
    rev: HEAD
    sshKeySecret: airflow-git-ssh-secret

# DB 연동 (외부 PostgreSQL 사용)
postgresql:
  enabled: false  # 내부 Postgres 비활성화
data:
  metadataSecretName: airflow-db-secret

# Airflow 환경 설정
config:
  AIRFLOW__CORE__LOAD_EXAMPLES:
    value: "False"
  AIRFLOW__CORE__DAGS_ARE_PAUSED_AT_CREATION:
    value: "False"

# AWS 인증 마운트 (scheduler/workers/dagProcessor 공통)
scheduler:
  extraVolumes:
    - name: aws-credentials
      secret:
        secretName: aws-credentials-secret
  extraVolumeMounts:
    - name: aws-credentials
      mountPath: /home/airflow/.aws
      readOnly: true

# 로그 PVC 설정
logs:
  persistence:
    enabled: true
    existingClaim: airflow-logs-pvc
```

> 전체 코드: [GitHub (values.yaml)](https://github.com/keonhoban/mlops-infra-gitops/tree/main/charts/airflow)

workers, dagProcessor에도 동일한 `extraVolumes`/`extraVolumeMounts` 설정이 필요합니다. Ingress는 `apiServer` 기반으로 `airflow.local` 호스트를 설정합니다.

## 🔑 Fernet Key 생성 및 구성

```bash
# 생성
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-fernet-secret
  namespace: airflow
type: Opaque
stringData:
  fernet-key: ""  # 생성한 값으로 수정
```

# SSH 키 생성

## 설계 판단 (Why This Way?)

- **KubernetesExecutor 선택 이유**: CeleryExecutor는 상시 Worker Pod를 유지하므로 리소스 낭비가 발생합니다. KubernetesExecutor는 Task마다 Pod를 생성/삭제하여 리소스를 효율적으로 사용하며, 로컬 환경에서 특히 유리합니다.
- **GitSync의 depth/wait/rev 튜닝**: `depth: 1`은 최신 커밋만 가져와 clone 속도를 높이고, `wait: 10`은 10초마다 변경사항을 확인합니다. `rev: HEAD`는 항상 최신 커밋을 추적합니다.
- **Fernet Key 고정의 의미**: Helm delete/install 시 Fernet Key가 재생성되면 기존 Variables, Connections의 암호화된 값을 복호화할 수 없습니다. Key를 Secret으로 고정하면 재배포 후에도 데이터가 유지됩니다.
- **PVC 로그 vs S3 로그**: PVC 방식은 설정이 간단하고 로컬 환경에 적합합니다. S3 로그는 영구 저장과 중앙 집중 관리에 유리하지만 네트워크 지연과 추가 설정이 필요합니다. 이 실습에서는 PVC를 사용하고, 프로덕션에서는 S3 전환을 권장합니다.
- **각 컴포넌트별 AWS 마운트**: scheduler, workers, dagProcessor 모두 독립 Pod이므로 각각 `.aws/credentials`를 마운트해야 합니다. 하나라도 누락되면 해당 컴포넌트에서 S3 접근이 실패합니다.

## 면접 포인트

- **Q: KubernetesExecutor와 CeleryExecutor의 차이는?**
  → CeleryExecutor는 상시 Worker Pod + 메시지 브로커(Redis/RabbitMQ)가 필요하고, KubernetesExecutor는 Task당 Pod를 동적 생성합니다. 리소스 효율성은 KubernetesExecutor가 우수하지만, Task 시작 지연(Pod 생성 시간)이 있습니다.

- **Q: GitSync의 동작 방식과 한계는?**
  → sidecar 컨테이너가 주기적으로 Git pull하여 DAG 디렉토리를 동기화합니다. 한계로는 branch 전환 시 동기화 지연, SSH 인증 실패 시 무한 재시도, 대규모 저장소에서의 네트워크 부하가 있습니다.

- **Q: Airflow에서 Fernet Key의 역할은?**
  → Variables, Connections 등 민감한 데이터를 메타데이터 DB에 저장할 때 Fernet 대칭키로 암호화합니다. Key가 변경되면 기존 암호화된 데이터를 복호화할 수 없습니다.
