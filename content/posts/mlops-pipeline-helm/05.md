## 📄 app/main.py (핵심 부분)

```python
app = FastAPI()
model = None
model_info = {}

def load_model_from_mlflow():
    global model, model_info
    tracking_uri = os.environ.get("MLFLOW_TRACKING_URI")
    model_name = os.environ.get("MODEL_NAME")
    model_stage = os.environ.get("MODEL_STAGE", "Production")

    mlflow.set_tracking_uri(tracking_uri)
    model_uri = f"models:/{model_name}/{model_stage}"
    model = mlflow.pyfunc.load_model(model_uri)

    client = MlflowClient()
    latest = client.get_latest_versions(name=model_name, stages=[model_stage])[0]
    model_info = {
        "model_name": model_name,
        "stage": model_stage,
        "version": latest.version,
        "run_id": latest.run_id,
        "model_uri": model_uri,
    }

@app.on_event("startup")
def startup_event():
    load_model_from_mlflow()

@app.get("/model-info")
def get_model_info():
    return model_info

@app.post("/predict")
async def predict(request: Request):
    input_data = await request.json()
    prediction = model.predict(input_data)
    return {"prediction": prediction.tolist()}
```

> 전체 코드: [GitHub (main.py)](https://github.com/keonhoban/mlops-infra-gitops/tree/main/charts/fastapi)

## 🛠 Helm 배포 구성

### values.yaml

```yaml
replicaCount: 1

image:
  repository: hoizz/fastapi-ml
  tag: mlflow-model-info
  pullPolicy: IfNotPresent

service:
  name: fastapi-service
  type: ClusterIP
  port: 80

ingress:
  enabled: true
  className: nginx
  host: fastapi.local
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /

env:
  MLFLOW_TRACKING_URI: "http://mlflow-service.mlflow.svc.cluster.local:5000"
  MODEL_NAME: "best_model"
  MODEL_STAGE: "Staging"

envFrom:
  - secretRef:
      name: aws-credentials-secret
```

## 설계 판단 (Why This Way?)

- **FastAPI vs Flask vs Triton**: FastAPI는 async 지원, 자동 API 문서 생성, 타입 검증이 내장되어 있어 빠른 프로토타이핑에 적합합니다. Triton은 고성능 서빙(GPU, 배치 처리)에 특화되지만 설정이 복잡합니다. 이 단계에서는 FastAPI로 핫스왑 구조를 먼저 검증합니다.
- **Stage 기반 모델 로딩**: `models:/{name}/{stage}` URI를 사용하면 MLflow에서 Stage만 전환하면 FastAPI가 다음 reload 시 새 모델을 로딩합니다. 버전 번호를 하드코딩하지 않아 운영 자동화에 유리합니다.
- **global model 변수의 한계**: `load_model_from_mlflow()`가 전역 변수 `model`을 교체하는 방식이므로, 교체 중 요청이 들어오면 일시적으로 불안정할 수 있습니다. 프로덕션에서는 락(lock) 메커니즘이나 Blue-Green 패턴이 필요합니다.
- **`/model-info` API 제공 이유**: 운영 중 현재 서빙 중인 모델의 버전, Stage, run_id를 확인할 수 있어야 합니다. 모델 핫스왑 후 실제로 교체가 됐는지 검증하는 용도입니다.
- **모델 미존재 시 예외 처리**: `load_model` 시 `MlflowException`이 발생하면 FastAPI가 죽지 않도록 예외 처리가 필요합니다. 배포 시 CPU/RAM 제한 설정으로 예측 지연도 방지해야 합니다.

## 면접 포인트

- **Q: 모델 핫스왑 시 요청 처리 중 교체가 일어나면?**
  → 전역 변수 교체 방식에서는 race condition이 발생할 수 있습니다. 프로덕션에서는 모델을 새 변수에 로드한 후 원자적으로 교체하거나, Blue-Green 패턴으로 새 Pod에 로드 후 트래픽을 전환합니다.

- **Q: MLflow Model Registry의 Stage 관리 방식은?**
  → None → Staging → Production → Archived 순서로 Stage를 전환합니다. `transition_model_version_stage()` API로 프로그래밍적으로 전환하며, 동일 Stage에 하나의 버전만 활성화됩니다.

- **Q: FastAPI의 `on_event("startup")`에서 모델을 로딩하는 이유는?**
  → 서버 시작 시 모델을 미리 메모리에 로딩하여 첫 번째 요청부터 즉시 추론할 수 있습니다. 요청 시점에 로딩하면 cold start 지연이 발생합니다.
