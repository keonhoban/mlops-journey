## 이 단계에서 해결하려는 문제

Deployment, Service, ConfigMap 등 여러 YAML 파일을 개별 관리하면, 환경별 설정 변경과 버전 관리가 어렵습니다. Helm은 이 YAML들을 하나의 Chart로 패키징하고, values.yaml로 설정만 바꿔서 재배포할 수 있게 합니다. MLOps에서 MLflow, ArgoCD, Prometheus 등을 설치할 때 사실상 표준 도구입니다.

실습 코드: [GitHub (Helm)](https://github.com/keonhoban/mlops-infra-labs/tree/main/k8s-basic/04_helm)


## ⚙️ [1단계] Helm 설치

```bash
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
helm version
```

---

## 📦 [2단계] Chart 저장소 등록 & 검색

```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
helm search repo nginx
```

> Helm Chart는 패키지화된 K8s 앱 템플릿입니다.

---

## 🚀 [3단계] Nginx 기본 설치 (LoadBalancer 방식)

```bash
helm install my-nginx bitnami/nginx
kubectl get all
```

```bash
# 결과 (일부)
service/my-nginx  LoadBalancer  10.103.5.39  <pending>  80:31794/TCP,443:32036/TCP
```

> LoadBalancer는 클라우드용 방식이라 Minikube에서는 `<pending>` 상태로 외부 접속이 안됩니다.

---

## 🛠️ [4단계] values.yaml로 NodePort 설정 변경

### values-nginx.yaml

```yaml
service:
  type: NodePort
  nodePorts:
    http: 30090
```

### 재설치

```bash
helm uninstall my-nginx
helm install my-nginx bitnami/nginx -f values-nginx.yaml
kubectl get all
```

```bash
# 결과 (정상 동작)
service/my-nginx  NodePort  10.109.159.213  <none>  80:30090/TCP,443:32694/TCP
```

### 접속 테스트

```bash
minikube ip         # IP 확인
curl http://<minikube-ip>:30090
```

## 📌 [5단계] 주요 Helm 명령어 정리

| 명령어 | 설명 |
| --- | --- |
| `helm list` | 설치된 Helm 릴리즈 목록 확인 |
| `helm uninstall <이름>` | Helm 배포 삭제 |
| `helm get all <이름>` | 생성된 모든 리소스 상세 보기 |
| `helm upgrade` | 설정값 변경 후 재배포 (`-f` 사용) |

```bash
helm upgrade my-nginx bitnami/nginx -f values-nginx.yaml
```

## 🎯 정리 요약

| 항목 | 설명 |
| --- | --- |
| Helm | K8s 앱을 패키징/설치/관리하는 도구 |
| Chart | 여러 YAML 파일과 설정값을 가진 패키지 템플릿 |
| values.yaml | 사용자 정의 설정값 (YAML 템플릿의 변수 대체용) |
| 기본 설치 결과 | LoadBalancer로 인해 Minikube에서는 외부 접속 어려움 |
| NodePort 설정 | 커스터마이징으로 실습 환경에 맞게 조정 가능 |

## 설계 판단 (Why This Way?)

- **Helm vs 순수 YAML 관리**: 소규모 프로젝트에서는 YAML 직접 관리도 가능하지만, MLOps 스택(MLflow, Airflow, Prometheus 등)은 수십 개의 리소스가 필요해 Helm 없이는 관리가 비현실적입니다.
- **values.yaml로 환경 분리**: 동일 Chart에 dev/staging/prod별 values 파일만 바꾸면 됩니다. GitOps(ArgoCD)와 결합하면 Git 커밋만으로 배포를 제어할 수 있습니다.
- **`helm upgrade` vs `uninstall + install`**: upgrade가 무중단 배포에 적합합니다. 이 실습에서는 학습 목적으로 uninstall → install 방식을 사용했습니다.
- **`--dry-run` 옵션**: 리소스 생성 전에 결과를 미리 확인할 수 있어, 프로덕션에서 안전한 변경 검증에 유용합니다.

## 면접 포인트

- **Q: Helm Chart와 Docker Image의 관계는?**
  → Docker Image는 컨테이너 실행 단위, Helm Chart는 그 Image를 포함한 K8s 리소스 전체(Deployment, Service, ConfigMap 등)를 패키징한 것입니다.

- **Q: values.yaml을 Git으로 관리하는 이유는?**
  → 환경별 설정을 코드로 추적하고, ArgoCD 같은 GitOps 도구가 Git 변경을 감지해 자동 배포합니다.

- **Q: Helm rollback은 어떻게 동작하나?**
  → `helm rollback <release> <revision>`으로 이전 릴리즈 상태로 복원합니다. Helm이 릴리즈 히스토리를 Secret으로 저장합니다.
