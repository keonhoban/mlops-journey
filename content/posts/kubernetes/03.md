## 이 단계에서 해결하려는 문제

Service마다 NodePort를 열면 포트 관리가 복잡해지고, 도메인 기반 접근이 불가능합니다. Ingress는 단일 진입점에서 HTTP 요청을 경로나 호스트 기반으로 내부 서비스에 라우팅합니다. 실무에서 MLflow, Jupyter, API 서버를 하나의 도메인 아래 경로로 분리할 때 필수적인 구조입니다.

실습 코드: [GitHub (Ingress_and_Nginx_Controller)](https://github.com/keonhoban/mlops-infra-labs/tree/main/k8s-basic/03_Ingress_and_Nginx_Controller)

## ⚙️ [1단계] Ingress Controller 설치

```bash
minikube addons enable ingress
kubectl get pods -n ingress-nginx
```

- `ingress-nginx-controller` Pod가 `Running` 상태면 설치 성공
- NodePort 기본 포트: `80`, `443`

## 🧱 [2단계] 서비스 2개 배포 (nginx / httpd)

### deploy-nginx.yaml (핵심 부분)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
```

> deploy-httpd.yaml도 동일 구조 (image: httpd, name: httpd-*).
> 전체 코드: [GitHub (Ingress_and_Nginx_Controller)](https://github.com/keonhoban/mlops-infra-labs/tree/main/k8s-basic/03_Ingress_and_Nginx_Controller)

## 🗺️ [3단계] Ingress 라우팅 설정

### ingress.yaml

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: example-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: foo.local
    http:
      paths:
      - path: /nginx
        pathType: Prefix
        backend:
          service:
            name: nginx-service
            port:
              number: 80
      - path: /httpd
        pathType: Prefix
        backend:
          service:
            name: httpd-service
            port:
              number: 80
```

## 🚀 [4단계] 실행 및 테스트

```bash
kubectl apply -f deploy-nginx.yaml
kubectl apply -f deploy-httpd.yaml
kubectl apply -f ingress.yaml
```

```bash
minikube ip
# 예: 192.168.49.2

sudo nano /etc/hosts
# 아래 줄 추가
192.168.49.2   foo.local
```

```bash
curl http://foo.local/nginx    # nginx 화면 반환
curl http://foo.local/httpd    # httpd 페이지 반환
```

## 🎯 정리 요약

| 항목 | 내용 |
| --- | --- |
| Ingress | HTTP(S) 요청을 Path/Host 기반으로 내부 서비스에 라우팅 |
| Ingress Controller | 실제 라우팅 처리 담당 (Nginx 등) |
| /etc/hosts | 로컬 DNS 역할 (클러스터 외부 접근 가능하게 설정) |
| 활용 사례 | `/api`, `/mlflow`, `/jupyter` 등 URL별 서비스 분리 가능 |

---

## 설계 판단 (Why This Way?)

서비스별 NodePort는 포트 충돌과 관리 복잡도가 높으므로, Ingress로 단일 포트에서 경로 기반 L7 라우팅을 구현한다. Nginx Ingress Controller가 MLOps 스택에서 가장 표준적인 선택이다.
