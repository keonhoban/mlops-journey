[GitOps 기반 E2E ML Platform - 검증]

March 6, 2026 · 8 min

이 글은 GitOps 기반 E2E ML Platform이

구조적으로 분리되어 있고, 실제로 동작하며, 운영 관측과 복구까지 가능한 상태임을

캡처와 실행 결과로 검증하는 기록입니다.

0) 🧰 준비#

이번 검증은 다음 기준으로 진행했습니다.

  • GitOps 구조는 ArgoCD Project / Application / Root App 기준으로 확인
  • Optional 레이어는 ON / OFF 실행 결과로 확인
  • Airflow 파이프라인은 E2E DAG orchestration 실행 결과 기준으로 확인
  • 서빙 상태는 Triton READY / FastAPI health / reload / metrics 기준으로 확인
  • 운영 관측은 Prometheus / Alertmanager / ServiceMonitor / PrometheusRule 기준으로 확인
  • 모든 결과는 docs/proof/latest/ 기준 최신 스냅샷으로 정리

사용한 주요 경로:

docs/proof/latest/projects.txt
docs/proof/latest/apps.txt
docs/proof/latest/root-apps.txt
docs/proof/latest/core_only/*
docs/proof/latest/optional_on/*

1) 🧭 GitOps 경계 검증#

1-1) ArgoCD Project 경계 검증#

📸 proof-01-projects.png

✅ 확인 포인트:

  • AppProject가 baseline, bootstrap, dev, optional, prod 로 분리되어 있으며

    project(default)는 기본 제공 프로젝트로 존재

  • dev는 -dev, prod는 -prod namespace 범위를 대상으로 함

  • optional은 feature-store / feast 전용 레이어로 분리됨

  • 각 AppProject는 GitOps repository

    (mlops-infra-gitops)를 주요 source로 사용하며,

    일부 baseline 컴포넌트는 Helm chart repository를 함께 사용

🧩 의미:

단순 namespace 이름 구분이 아니라,

ArgoCD AppProject 수준에서 배포 경계를 강제하고 있음을 증명합니다.

즉 dev / prod / optional / baseline이 구조적으로 분리된 GitOps 환경입니다.

1-2) ArgoCD Application 전체 구조 검증#

📸 proof-02-01-apps-list.png

📸 proof-02-02-apps-list.png

4) 🔄 Airflow E2E DAG 실행 검증#

4-1) E2E DAG 전체 성공 흐름 검증#

📸 proof-09-01-e2e-dag-success.png

📸 proof-09-02-e2e-dag-success.png

✅ 확인 포인트:

  • e2e_full DAG에서 dp.extract_raw_data → dp.validate_data → dp.build_features → dp.store_features → drift_gate → train → branch → register → sensor_model_ready → deploy → commit → fastapi_reload → observe → summarize_run 흐름이 성공
  • shadow_start, notify_failure, rollback_minimal 태스크는 이번 실행에서 분기 조건상 skipped 처리됨
  • dp, deploy TaskGroup 포함 전체 orchestration chain이 정상 종료됨

🧩 의미:

이 플랫폼이 단순히 Triton/FastAPI가 떠 있는 구조가 아니라,

Airflow가 데이터 준비부터 운영 반영 후 관측까지 전체 흐름을 제어하는 E2E ML Platform 임을 증명합니다.

4-2) Promotion / Shadow 분기 구조 검증#

📸 proof-09-03-branch-promotion-shadow.png

📸 proof-09-04-branch-promotion-shadow.png

✅ 확인 포인트:

  • check_result 이후 정책 및 임계값에 따라 promotion_start 또는 shadow_start 로 분기
  • 성공 사례에서는 register_model_task → check_model_ready → deploy 흐름으로 이어져 MLflow register 및 FastAPI reload까지 진행
  • 차단 사례에서는 shadow_start 와 notify_failure 흐름이 선택되며, Slack 알림에 Branch: shadow, reason: below_threshold, Shadow path selected 메시지가 남음

🧩 의미:

이 플랫폼은 학습 완료를 곧바로 운영 반영으로 보지 않고,

임계값 및 정책 기반으로 promotion / shadow 경로를 실제로 분기하는 운영형 ML 파이프라인임을 증명합니다.

4-3) READY Sensor 배포 게이트 검증#

📸 proof-09-05-model-ready-sensor.png

✅ 확인 포인트:

  • promotion 경로 실행에서는 check_model_ready 태스크가 register 이후, deploy 이전 단계에서 성공
  • shadow 분기 실행에서는 check_model_ready 가 skipped 처리되어, 배포 게이트가 promotion 경로에서만 적용됨
  • 모델이 READY 상태가 되기 전까지 deploy 단계가 진행되지 않음

🧩 의미:

등록된 모델이 존재한다고 바로 운영에 반영하는 것이 아니라,

READY sensor를 명시적 배포 게이트로 사용하는 안전한 운영 반영 구조임을 증명합니다.

4-4) Manual Rollback DAG 실행 검증#

📸 proof-09-06-manual-rollback-dag.png

📸 proof-09-07-manual-rollback-dag.png

✅ 확인 포인트:

  • rollback_manual DAG에서 load_ctx → guard_repo → write_ssot_files → rollback_triton → reload_fastapi → verify_convergence 전 단계 성공
  • rollback 이후 /models 응답에서 SSOT 기준 served version이 이전 버전으로 복원되고, Slack에도 FastAPI reload 결과가 기록됨
  • clear pod-local cache 수행 후 cache_pod_local 이 비워진 상태로 확인되어, pod-local override가 아닌 SSOT 기준 상태로 수렴했음을 확인

🧩 의미:

자동 rollback과 별개로,

운영자가 명시적으로 복구를 수행할 수 있는 수동 롤백 가드레일이 실제 DAG로 구현되어 있음을 증명합니다.
