[Feature Store & Feast - 에필로그]

January 16, 2026 · 2 min

Feature Store는 ‘피처 생성’이 아니라 운영 재현성의 기준#

📌 시리즈 전체 경로 요약#

순서주제
1Contract 기반 Feature Store-lite 구축
2Feast 확장으로 조회 가능한 플랫폼 완성
3Proof로 운영 신뢰성 검증

이 시리즈는 “Feast를 써봤다”가 아니라,

Feature Store를 운영 가능한 시스템으로 고정하는 과정을 기록했습니다.

🎯 전체 회고 요약#

단계목표고정된 기준
1Feature Store-lite 뼈대 구축Contract 분리 + 버전 1단 + 재현성 메타
2Feast로 플랫폼 확장version + latest 정책 + offline/online 일관성
3Proof로 신뢰성 확보E2E 검증 + 장애 재현·복구

🔄 이 시리즈를 관통하는 한 문장#

Contract → Build → Versioned Store → Latest Pointer → Apply → Materialize → Online Read

Feature Store를

  • 파일 생성 파이프라인이 아니라
  • *운영 기준이 고정된 상태 머신(state machine)**으로 정의했고

모든 결과가 재현 가능한 메타데이터로 남도록 설계했습니다.

🧩 Feature Store 전체 구조 (Lite → Feast)#

구조 요약#

  • Contract 계층

    schema / repo를 GitOps 리소스로 고정

  • Control 계층

    Airflow는 “생성 + 저장 정책”까지만 담당

  • Storage 계층

    version(재현성) + latest(운영 편의) 동시 유지

  • Platform 계층

    Feast가 offline / online / registry를 책임

  • dev/prod 분리 원칙

    namespace · prefix · application 단위 완전 분리

🔁 Feature Store Operational Loop#

  • GitOps로 계약이 먼저 고정되고
  • Airflow는 계약을 읽어 동일한 규칙으로 피처를 생성하며
  • S3에는 version과 latest가 함께 유지되고
  • Feast는 latest만 바라보며 offline/online 일관성을 보장합니다

모델 학습, 서빙, 모니터링은

feature version 또는 registry 기준으로 자연스럽게 확장됩니다.

🧠 운영 원칙 정리#

1️⃣ Contract First#

  • schema / repo는 코드가 아니라 운영 리소스
  • GitOps로 배포·롤백 가능해야 함
  • 파이프라인은 “읽기 전용 소비자”로 제한

2️⃣ Version은 깊어지지 않는다#

  • /<feature_set>/<version>/ 1단 고정
  • 재현성은 depth가 아니라 규칙에서 나온다

3️⃣ latest는 편의 장치일 뿐이다#

  • Feast offline은 latest만 사용
  • 재현/분석/학습은 언제든 version으로 회귀 가능

4️⃣ 실패는 삭제하지 않고 격리한다#

  • s3fs, subPath, Airflow Variable SDK 등

    실제로 깨졌던 지점은

    재현 → 해결 → 가드레일로 고정

  • 운영 시스템은 “깨졌던 흔적”을 남길수록 강해진다

✅ 최종 점검 체크리스트 (Proof 기반)#

  • ArgoCD에서 feature-store / feast dev·prod Synced/Healthy
  • Airflow Pod에서 계약 파일 mount 확인
  • DAG E2E 성공 (build → store)
  • S3에 version prefix + latest 동시 존재
  • metadata.json에 schema_hash / generated_at 기록
  • Feast apply 성공 + registry S3 반영
  • materialize 성공 (s3fs 포함 이미지)
  • online(redis) 조회 결과가 None 아님

🏁 회고#

이번 시리즈의 결론은

“Feature Store를 만들었다”가 아닙니다.

피처가 생성·저장·조회되는 전 과정을 ‘운영 기준’으로 고정했다는 점입니다.

Feature Store-lite에서는 재현성의 뼈대를 만들었고,

Feast 확장에서는 그 위에 조회 가능한 플랫폼을 올렸습니다.

특히, 실제로 깨졌던 포인트들을 회피하지 않고

재현 → 해결 → 구조 고정까지 가져갔습니다.

이제 이 Feature Store는

Training / Serving / Monitoring으로

자연스럽게 확장될 수 있는 안전한 기준점이 되었습니다.
