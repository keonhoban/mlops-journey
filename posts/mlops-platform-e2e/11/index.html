왜 부하 테스트를 포함시켰는가#

많은 ML 플랫폼 글은 다음 단계에서 검증을 마칩니다.

지속적인 트래픽 환경에서 안정적으로 동작해야 합니다.

그래서 이 프로젝트에서는

테스트 환경#

부하 테스트 환경은 다음과 같습니다.

클러스터: 3노드 Kubernetes (CPU-only)
FastAPI:  2 replica
Triton:   1 replica (CPU-only, dynamic_batching 적용)

테스트 대상 endpoint:

POST /predict

이 endpoint는 다음 경로로 요청을 처리합니다.

k6
 ↓
FastAPI /predict
 ↓
Triton /v2/models/best_model/infer
 ↓
응답 반환

즉

클라이언트 → FastAPI → Triton

전체 서빙 경로를 포함한 E2E 부하 테스트입니다.

환경 제약:

GPU 없음
CPU 추론만 사용

이 환경은 운영 환경 대비 제약된 환경입니다.

k6 스크립트 구조#

k6 스크립트는 다음 3단계로 구성됩니다.

Ramp-up → Sustained → Ramp-down

각 단계는 다음과 같습니다.

Ramp-up:    0 → 50 VU   (30초)
Sustained:  100 VU 유지  (90초)
Ramp-down:  100 → 0 VU   (30초)

총 실행 시간: 150초

스크립트 예시:

import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 50 },
    { duration: '90s', target: 100 },
    { duration: '30s', target: 0 },
  ],
  thresholds: {
    http_req_failed:   ['rate<0.01'],
    http_req_duration: ['p(95)<2000'],
  },
};

const payload = JSON.stringify({
  data: [[5.1, 3.5, 1.4, 0.2]],
});

const params = {
  headers: { 'Content-Type': 'application/json' },
};

export default function () {
  const res = http.post('http://fastapi-dev/predict', payload, params);
  check(res, {
    'status 200': (r) => r.status === 200,
  });
  sleep(0.1);
}

이 구조는 다음 목적을 가집니다.

Ramp-up:   점진적 부하 증가로 시스템 워밍업
Sustained: 최대 부하(100 VU)에서 처리량 및 에러율 측정
Ramp-down: 부하 해소 후 시스템 복구 확인

p95 결과 분석#

k6 스크립트에 설정한 p95 threshold는 다음입니다.

p(95) < 2000ms

실측 p95는 553ms로 threshold를 통과했습니다.

이 수치 자체를 분석하면 다음과 같습니다.

CPU-only 환경의 한계

Triton 추론 엔진은 GPU에 최적화
CPU 추론은 thread 기반 병렬 처리
CPU 코어 수 제한 → 배치 처리 병목

이 환경에서 p50 126ms는 합리적인 수치입니다.

하지만 p95가 553ms까지 오르는 것은

Peak 부하 구간에서 CPU 추론 큐가 쌓이는 것이 주된 원인입니다.

GPU 환경 예상 개선치

GPU 추론 환경에서는 다음이 기대됩니다.

p95: 553ms → 50ms 이하 (예상)
처리량: 136 RPS → 수백 RPS 이상 (예상)

이는 다음 이유 때문입니다.

GPU 병렬 처리
dynamic_batching 효율 극대화
CUDA 기반 추론 가속

즉 현재 결과는

CPU-only 3노드 클러스터의 실제 한계를 보여주는 수치입니다.

HPA 트리거 기준과의 연결#

이 부하 테스트 결과는

HPA(Horizontal Pod Autoscaler) 설계와 연결됩니다.

이 프로젝트의 HPA 트리거 기준:

CPU 사용률 70% 초과 → 스케일 아웃
replica: 2 ~ 5

관련 설정:

# charts/fastapi/templates/hpa.yaml (향후 적용 예정 — 현재 미생성)
spec:
  minReplicas: 2
  maxReplicas: 5
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

참고: 이 HPA 템플릿은 아직 실제 Helm 차트에 생성되지 않은 상태입니다. 부하 테스트 결과를 바탕으로 향후 적용할 설계 기준을 정리한 것입니다.

부하 테스트 결과와의 연결:

136 RPS 구간에서 FastAPI CPU 사용률 증가
→ 70% threshold 도달 시 replica 3→4→5 자동 확장
→ 처리량 분산, latency 안정화

즉 이 설정은 다음을 의미합니다.

136 RPS = 2 replica 처리 가능한 적정 부하
100 VU 이상 지속 시 HPA 트리거 예상

설계 판단 (Why This Way?)#

  • 부하 테스트를 E2E 검증에 포함시킨 이유: 단일 요청 성공과 100명 동시 요청은 전혀 다른 문제다. API 호출 성공만으로는 운영 트래픽에서의 안정성을 보장할 수 없으므로 부하 테스트를 Proof의 일부로 포함했다.
  • Ramp-up → Sustained → Ramp-down 3단계 구성: 점진적 부하 증가로 시스템 워밍업을 하고, 최대 부하에서 처리량과 에러율을 측정하며, 부하 해소 후 시스템 복구를 확인한다. 실제 트래픽 패턴에 가까운 테스트 구조다.
  • HPA 설계 기준과 연결: 부하 테스트 결과를 바탕으로 HPA 트리거 기준(CPU 70%, replica 2~5)을 설정했다. 실측 데이터 없이 임의로 설정하면 과잉/부족 확장이 발생한다.

면접 포인트#

  • Q: CPU-only 환경에서 p95 553ms의 원인은? → Triton 추론 엔진은 GPU에 최적화되어 있다. CPU 환경에서는 thread 기반 병렬 처리로 코어 수 제한에 따른 배치 처리 병목이 발생한다. Peak 부하 구간에서 CPU 추론 큐가 쌓이면서 p95가 증가한다.

  • Q: 에러율 0%가 의미하는 것은? → 20,553건 전량 성공이므로 서빙 시스템의 안정성이 확인된다. 다만 이는 CPU-only 환경의 처리 한계(136 RPS) 내에서의 결과이며, 이 이상의 트래픽에서는 HPA를 통한 수평 확장이 필요하다.

  • Q: GPU 환경으로 전환하면 어떤 개선이 예상되는가? → GPU 병렬 처리와 dynamic_batching 효율 극대화로 p95 50ms 이하, 처리량 수백 RPS 이상이 기대된다.
