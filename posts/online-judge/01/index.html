[OJ] 문제 해결 사고 프레임 정리

December 8, 2025 · 2 min

들어가며#

문제 난이도가 올라가며 가장 답답했던 지점은 단순했습니다.

해설을 보면 이해되는데, 문제를 처음 마주하면 손이 잘 안 움직였습니다.

문제에 접근하는 ‘사고 구조’가 정리되어 있지 않았기 때문이었습니다.

이 글은 문제 풀이 기법이 아니라,

문제를 처음 마주했을 때 어디서부터 생각해야 하는지

스스로 정리하기 위해 적용해본 사고 프레임 기록입니다.

1️⃣ 왜 사고 프레임이 필요했는가#

문제를 풀다 보면 이런 상태가 반복되었습니다. (* 특히 여러 조건이 결합된 시물레이션 문제들)

  • 해설을 보면 “아, 그렇구나”
  • 며칠 뒤 다시 보면 혼자서는 못 풂
  • 문제를 많이 풀어도 체감 실력은 정체

이 상태를 벗어나기 위해

“문제를 푸는 것”보다 “문제를 다루는 방식”에 집중했습니다.

2️⃣ 문제 해결 5단계 사고 프레임#

① 문제 재정의 (핵심 요점)#

  • 입력 / 출력 / 종료 조건을 자연어로 다시 작성
  • 문제 설명을 코드가 아닌 문장으로 이해

② 상태(State) 분해#

  • 턴마다 변하는 값 / 변하지 않는 값 구분
  • 예: 좌표, 시간, 방문 여부, 누적 값

③ 탐색 전략 선택#

  • BFS / DFS / Heap / 투포인터 중 선택
  • “왜 이 방법이 맞는가?”를 말로 설명 가능해야 함

④ 종료 및 실패 조건 확인#

  • 언제 틀릴 수 있는지 먼저 생각
  • 예외 / 경계 / 종료 조건을 코드보다 먼저 정의

⑤ 코드 작성 후 사고 역추적#

  • 코드 → 사고 흐름이 맞는지 다시 검증
  • 한 줄마다 “이 줄이 필요한 이유”를 설명 가능해야 통과

3️⃣ 혼자 못 풀었던 문제를 다루는 방식#

솔직히 많은 문제를 처음에는 혼자 못 풀었습니다.

하지만 해설을 보는 순간이 끝이 아니라 시작이 되도록 규칙을 만들었습니다.

  • 해설을 본 뒤 빈 노트에 사고 흐름부터 작성
  • 코드를 바로 치지 않고, 단계별로 정리
  • 다시 문제를 가렸을 때 코드가 안 나오면 미완료 처리

이 기준을 통과하지 못한 문제는

“푼 문제”가 아니라 “보류 문제”로 남겼습니다.

4️⃣ 이 프레임을 유지하기 위한 개인 규칙#

  • 하루 최소 문제 수보다 완성 기준을 우선
  • “다시 풀 수 있음”이 체크 조건
  • 못 풀면 기록하고, 다시 돌아오기

마무리#

이 프레임은 아직 완성형이 아닙니다.

하지만 이제 문제를 무작정 푸는게 아니라, 설계부터 할 수 있게 되었습니다.
