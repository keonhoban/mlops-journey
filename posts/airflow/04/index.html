[Airflow - 4단계: BashOperator로 외부 Python 학습 스크립트 실행]

June 10, 2025 · 3 min

이 글에서 다루는 것#

DAG 내부에 로직을 넣지 않고, 별도 Python 학습 스크립트(train.py)를 BashOperator로 호출하여 코드와 오케스트레이션을 분리하는 패턴을 다룹니다.

선수지식#

  • Airflow 3단계: ML 파이프라인 DAG 구성 — DAG 구조와 XCom 전달

이 단계에서 해결하려는 문제#

이전 단계에서는 학습 로직을 DAG 파일 안에 직접 작성했습니다. 하지만 실무에서는 학습 코드가 수백 줄 이상이고, 데이터 사이언티스트가 별도로 관리합니다. DAG는 오케스트레이션만 담당하고, 학습 코드는 독립 스크립트로 분리해야 유지보수와 협업이 가능합니다.

실습 코드: GitHub (BashOperator_and_Python_ML_Script)

🧭 실습 전체 흐름 요약#

[1단계] 학습 스크립트 작성
[2단계] BashOperator로 스크립트 실행 DAG 구성
[3단계] Airflow 웹 UI에서 실행 및 로그 확인

📁 디렉토리 구조#

airflow/
├── dags/
│   └── run_train_script.py   ← DAG 파일
├── ml_code/
│   ├── train.py              ← 모델 학습 스크립트
│   └── model.pkl             ← 학습된 모델 파일

🧪 [1단계] 학습 스크립트 작성 (train.py)#

# airflow/ml_code/train.py

import pickle
from sklearn.datasets import load_iris
from sklearn.ensemble import RandomForestClassifier

data = load_iris()
X, y = data.data, data.target

model = RandomForestClassifier()
model.fit(X, y)

model_path = "/opt/airflow/ml_code/model.pkl"
with open(model_path, "wb") as f:
    pickle.dump(model, f)

모델은 /opt/airflow/ml_code/model.pkl 경로에 저장됩니다.

🧪 [2단계] DAG 작성 (run_train_script.py)#

# airflow/dags/run_train_script.py

from airflow import DAG
from airflow.operators.bash import BashOperator
from datetime import datetime

with DAG(
    dag_id='bash_run_train',
    start_date=datetime(2023, 1, 1),
    schedule_interval=None,
    catchup=False,
) as dag:

    run_training = BashOperator(
        task_id='run_train_script',
        bash_command='python3 /opt/airflow/ml_code/train.py'
    )

train.py는 Airflow 컨테이너 내 /opt/airflow/ml_code/ 경로에 위치해야 합니다.

✅ [3단계] Airflow 컨테이너 진입 및 확인#

docker exec -it airflow-airflow-webserver-1 bash
cd /opt/airflow/ml_code
ls -l

train.py와 model.pkl 파일이 보이면 성공

설계 판단 (Why This Way?)#

  • BashOperator vs PythonOperator: 외부 스크립트를 그대로 실행할 때는 BashOperator가 적합합니다. DAG 파일에 import하지 않으므로 의존성 충돌 위험이 줄어듭니다.
  • 코드 분리 (DAG vs ml_code): DAG는 “언제, 무엇을” 실행할지만 정의하고, “어떻게"는 별도 스크립트에 둡니다. 이 분리가 팀 협업의 핵심입니다.
  • Volume 마운트: docker-compose.yaml에서 ./ml_code:/opt/airflow/ml_code로 마운트하여 호스트의 스크립트가 컨테이너에서 접근 가능합니다.
  • 대안 — KubernetesPodOperator: 프로덕션에서는 학습 스크립트를 별도 컨테이너 이미지로 패키징하고, KubernetesPodOperator로 실행하는 것이 격리와 리소스 관리에 유리합니다.

면접 포인트#

  • Q: DAG에 학습 로직을 직접 넣지 않는 이유는? → DAG는 오케스트레이션 책임만 가져야 합니다. 학습 코드가 DAG에 들어가면 테스트·버전 관리·팀 협업이 어려워집니다.

  • Q: BashOperator로 스크립트를 실행할 때의 리스크는? → 스크립트 경로 하드코딩, 컨테이너 내 의존성 관리, 에러 발생 시 exit code 기반 판단만 가능합니다. PythonOperator나 KubernetesPodOperator가 더 세밀한 제어를 제공합니다.

  • Q: Airflow Worker에 ML 라이브러리를 설치하는 것이 좋은 방법인가? → Worker 이미지가 무거워지고 의존성 충돌 위험이 있습니다. KubernetesPodOperator로 학습 전용 이미지를 분리하는 것이 프로덕션 패턴입니다.
