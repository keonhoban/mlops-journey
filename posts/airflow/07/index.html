Airflow DAG CI: 파싱 오류를 운영 전에 잡는 구조#

🧠 시나리오 설명#

“DAG 파일에 오타가 하나 들어갔습니다.

Airflow Scheduler가 파싱 오류를 감지하고 해당 DAG을 비활성화합니다.

문제를 발견한 것은 배포 후 30분이 지난 시점입니다.”

“이 프로젝트에서는 이 상황을 방지하기 위해

DAG CI를 구성했습니다.

push 시점에 파싱 오류를 잡고, 운영 환경까지 가지 않도록 합니다.”

🎯 핵심 요약#

  • DAG CI 목적: 파싱 오류 / import 오류를 배포 전에 감지
  • test_dag_integrity.py: DagBag import, task 수 검증, SLA 설정 검증
  • GitHub Actions: push/PR 시 자동 실행, Python 환경에서 테스트
  • 로컬 실행: pytest tests/test_dag_integrity.py 로 동일하게 검증 가능
  • CI 실패 시: 오류 로그에서 파일명과 원인 즉시 확인 가능

1️⃣ 왜 DAG CI가 필요한가#

Airflow DAG 파일은 Python 코드입니다.

다음과 같은 오류가 흔히 발생합니다.

import 오류 (없는 모듈 참조)
SyntaxError (오타, 들여쓰기 오류)
DAG 정의 오류 (dag_id 중복, schedule 형식 오류)
Task 의존성 오류 (순환 참조)

이 오류들은 Airflow Scheduler가 실행하기 전까지 발견되지 않습니다.

DAG 코드 수정
      ↓
Git push
      ↓
ArgoCD sync → Airflow에 반영
      ↓
Scheduler가 DAG 파일 파싱
      ↓
파싱 오류 발견 → DAG 비활성화

이 흐름에서 문제를 발견하는 것은 배포 이후입니다.

파싱 오류 운영 중 발견 비용:

파이프라인 중단
Scheduler 에러 로그 확인
코드 수정 → 재배포
최소 수십 분 소요

DAG CI는 이 비용을 push 시점으로 앞당깁니다.

코드 수정
      ↓
Git push
      ↓
GitHub Actions → test_dag_integrity.py 실행
      ↓
파싱 오류 즉시 감지 → PR 차단

2️⃣ test_dag_integrity.py 구조#

관련 파일:

tests/test_dag_integrity.py

이 테스트 파일은 다음 세 가지를 검증합니다.

DagBag import 검증
Task 수 검증
SLA 설정 검증

(1) DagBag import 검증#

import pytest
from airflow.models import DagBag

DAG_FOLDER = "dags"

@pytest.fixture(scope="module")
def dagbag():
    return DagBag(dag_folder=DAG_FOLDER, include_examples=False)


def test_no_import_errors(dagbag):
    """DAG 파일 import 오류가 없어야 한다."""
    assert dagbag.import_errors == {}, (
        f"DAG import errors: {dagbag.import_errors}"
    )

DagBag은 Airflow가 DAG 파일을 로드하는 방식과 동일합니다.

import_errors가 비어 있지 않으면 테스트가 실패합니다.

오류 메시지에서 파일명과 원인을 바로 확인할 수 있습니다.

AssertionError: DAG import errors: {
  'dags/mlops_train.py': 'No module named mlops_lib.feature'
}

(2) Task 수 검증#

def test_dag_task_count(dagbag):
    """각 DAG에 최소 1개 이상의 Task가 있어야 한다."""
    for dag_id, dag in dagbag.dags.items():
        assert len(dag.tasks) > 0, (
            f"DAG '{dag_id}' has no tasks"
        )

Task가 0개인 DAG은 잘못 정의된 DAG입니다.

이 검증으로 빈 DAG이 배포되는 것을 방지합니다.

(3) SLA 설정 검증#

SLA_REQUIRED_DAGS = [
    "mlops_train_dev",
    "mlops_deploy_dev",
    "mlops_train_prod",
    "mlops_deploy_prod",
]


def test_dag_sla_set(dagbag):
    """운영 DAG에는 SLA(sla_miss_callback)가 설정되어 있어야 한다."""
    for dag_id in SLA_REQUIRED_DAGS:
        if dag_id not in dagbag.dags:
            continue
        dag = dagbag.dags[dag_id]
        assert dag.sla_miss_callback is not None, (
            f"DAG '{dag_id}' is missing sla_miss_callback"
        )

운영 DAG에 SLA가 빠지면

장애 감지가 늦어집니다.

이 검증으로 SLA 누락을 배포 전에 잡습니다.

3️⃣ GitHub Actions workflow 구조#

관련 파일:

.github/workflows/ci.yml

전체 구조:

name: DAG Integrity Test

on:
  push:
    branches:
      - main
      - dev
    paths:
      - 'dags/**'
      - 'tests/**'
  pull_request:
    branches:
      - main
      - dev
    paths:
      - 'dags/**'
      - 'tests/**'

jobs:
  dag-integrity:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.10'

      - name: Install dependencies
        run: |
          pip install apache-airflow==2.9.0 pytest

      - name: Run DAG integrity tests
        env:
          AIRFLOW__CORE__LOAD_EXAMPLES: 'False'
          AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: 'sqlite:////tmp/airflow_test.db'
        run: |
          airflow db migrate
          pytest tests/test_dag_integrity.py -v

주요 설계 포인트#

paths 필터:

paths:
  - 'dags/**'
  - 'tests/**'

DAG 파일 또는 테스트 파일이 변경될 때만 CI가 실행됩니다.

README 수정 등 불필요한 실행을 방지합니다.

SQLite DB 사용:

AIRFLOW__DATABASE__SQL_ALCHEMY_CONN: 'sqlite:////tmp/airflow_test.db'

CI 환경에서 PostgreSQL 없이 SQLite로 Airflow DB를 구성합니다.

airflow db migrate로 스키마를 초기화한 후 테스트를 실행합니다.

LOAD_EXAMPLES 비활성화:

AIRFLOW__CORE__LOAD_EXAMPLES: 'False'

Airflow 기본 예시 DAG가 로드되지 않도록 합니다.

테스트 범위를 프로젝트 DAG만으로 한정합니다.

4️⃣ CI 실패 시나리오와 대응#

시나리오 1: import 오류#

# dags/mlops_train.py
from mlops_lib.feature import build_features  # 존재하지 않는 모듈

CI 실패 메시지:

FAILED tests/test_dag_integrity.py::test_no_import_errors
AssertionError: DAG import errors: {
  'dags/mlops_train.py': "No module named 'mlops_lib.feature'"
}

대응:

1. 오류 메시지에서 파일명과 모듈명 확인
2. 해당 파일에서 import 경로 수정
3. 로컬에서 pytest 재실행 후 push

시나리오 2: SyntaxError#

# dags/mlops_deploy.py
def train_task()
    pass  # 콜론 누락

CI 실패 메시지:

FAILED tests/test_dag_integrity.py::test_no_import_errors
AssertionError: DAG import errors: {
  'dags/mlops_deploy.py': "invalid syntax (mlops_deploy.py, line 5)"
}

대응:

1. 오류 메시지에서 파일명과 라인 번호 확인
2. 해당 라인 수정
3. 로컬에서 pytest 재실행

시나리오 3: SLA 누락#

FAILED tests/test_dag_integrity.py::test_dag_sla_set
AssertionError: DAG 'mlops_train_prod' is missing sla_miss_callback

대응:

# DAG 정의에 sla_miss_callback 추가
from mlops_lib.alert import notify_sla_miss

with DAG(
    dag_id="mlops_train_prod",
    sla_miss_callback=notify_sla_miss,
    ...
) as dag:
    ...

5️⃣ 로컬 실행 방법#

CI와 동일한 환경을 로컬에서도 실행할 수 있습니다.

# 의존성 설치
pip install apache-airflow==2.9.0 pytest

# DB 초기화
export AIRFLOW__CORE__LOAD_EXAMPLES=False
export AIRFLOW__DATABASE__SQL_ALCHEMY_CONN=sqlite:////tmp/airflow_test.db
airflow db migrate

# 테스트 실행
pytest tests/test_dag_integrity.py -v

정상 실행 결과:

tests/test_dag_integrity.py::test_no_import_errors PASSED
tests/test_dag_integrity.py::test_dag_task_count PASSED
tests/test_dag_integrity.py::test_dag_sla_set PASSED

3 passed in 4.21s

push 전에 로컬에서 먼저 실행하는 것이 권장 흐름입니다.

코드 수정
      ↓
pytest tests/test_dag_integrity.py -v  (로컬)
      ↓
PASSED 확인
      ↓
git push
      ↓
GitHub Actions 재확인

6️⃣ 실제 코드 전체#

tests/test_dag_integrity.py#

import pytest
from airflow.models import DagBag

DAG_FOLDER = "dags"

SLA_REQUIRED_DAGS = [
    "mlops_train_dev",
    "mlops_deploy_dev",
    "mlops_train_prod",
    "mlops_deploy_prod",
]


@pytest.fixture(scope="module")
def dagbag():
    return DagBag(dag_folder=DAG_FOLDER, include_examples=False)


def test_no_import_errors(dagbag):
    """DAG 파일 import 오류가 없어야 한다."""
    assert dagbag.import_errors == {}, (
        f"DAG import errors: {dagbag.import_errors}"
    )


def test_dag_task_count(dagbag):
    """각 DAG에 최소 1개 이상의 Task가 있어야 한다."""
    for dag_id, dag in dagbag.dags.items():
        assert len(dag.tasks) > 0, (
            f"DAG '{dag_id}' has no tasks"
        )


def test_dag_sla_set(dagbag):
    """운영 DAG에는 SLA(sla_miss_callback)가 설정되어 있어야 한다."""
    for dag_id in SLA_REQUIRED_DAGS:
        if dag_id not in dagbag.dags:
            continue
        dag = dagbag.dags[dag_id]
        assert dag.sla_miss_callback is not None, (
            f"DAG '{dag_id}' is missing sla_miss_callback"
        )

🧩 팁#

  • DagBag은 Airflow Scheduler가 DAG을 로드하는 방식과 동일합니다. CI에서 같은 방식으로 검증하면 운영과의 차이가 없습니다.
  • include_examples=False를 반드시 설정해야 Airflow 기본 DAG이 테스트에 포함되지 않습니다.
  • CI 환경에서 AIRFLOW__CORE__DAGS_FOLDER를 DAG_FOLDER와 일치시키면 더 정확한 검증이 가능합니다.
  • SLA 검증 대상 목록(SLA_REQUIRED_DAGS)은 운영 DAG이 추가될 때 함께 관리해야 합니다.

🏁 정리#

검증 항목테스트 함수감지 오류
import 오류test_no_import_errorsSyntaxError, ModuleNotFoundError 등
Task 수test_dag_task_count빈 DAG 정의
SLA 설정test_dag_sla_set운영 DAG SLA 누락

DAG CI는 파싱 오류를 운영 환경 배포 전에 잡습니다.

DagBag으로 실제 로드 방식과 동일하게 검증하고,

GitHub Actions로 push 시점에 자동 실행합니다.

로컬에서도 pytest tests/test_dag_integrity.py -v 하나로 동일하게 확인할 수 있습니다.
