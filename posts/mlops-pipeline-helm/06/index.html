[MLOps 플랫폼 구축 - 6단계: 실시간 모델 핫스왑 구조 실험]

July 10, 2025 · 3 min

이 글에서 다루는 것#

Airflow DAG에서 조건부 모델 등록 후 FastAPI 핫스왑까지 E2E 자동화 흐름을 실험합니다.

선수지식#

  • MLOps 플랫폼 구축 5단계: FastAPI 서빙 및 핫스왑 구조 구축 — FastAPI 모델 로딩과 /reload API 구조

이 단계에서 해결하려는 문제#

개별 컴포넌트(Airflow, MLflow, FastAPI)를 각각 구성했지만, 학습부터 서빙까지의 자동화 흐름을 E2E로 검증해야 실제 운영 가능 여부를 판단할 수 있습니다. 이 단계에서는 Airflow DAG에서 모델 학습 → 성능 기준 분기 → MLflow 등록 → FastAPI 핫스왑까지 전체 파이프라인을 실험합니다.

🧠 구조 다이어그램#

🧩 핵심 구성 요소#

1. Airflow DAG - 조건부 모델 등록 (핵심 로직)#

def run_and_check():
    # Variable로 파라미터 동적 제어
    try:
        C = float(Variable.get("logreg_C", default_var=1.0))
        max_iter = int(Variable.get("logreg_max_iter", default_var=200))
    except Exception as e:
        C, max_iter = 1.0, 200

    # 정확도 기준으로 성공/실패 분기
    acc = train_model(C=C, max_iter=max_iter)
    if acc > 0.9:
        return 'notify_success'
    else:
        return 'notify_failure'

with DAG(
    dag_id="mlflow_experiment_conditional_register_runner_fastapi",
    schedule=None,
    catchup=False,
) as dag:
    branch_task = BranchPythonOperator(
        task_id='branch_by_accuracy',
        python_callable=run_and_check
    )
    branch_task >> [success_notify_task, failure_notify_task]

2. 모델 학습 및 등록 (핵심 로직)#

def train_model(C, max_iter):
    X, y = load_iris(return_X_y=True)
    X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

    with mlflow.start_run() as run:
        clf = LogisticRegression(C=C, max_iter=max_iter)
        clf.fit(X_train, y_train)
        acc = accuracy_score(y_test, clf.predict(X_test))

        mlflow.log_param("C", C)
        mlflow.log_metric("accuracy", acc)
        mlflow.sklearn.log_model(clf, "model")

        if acc > 0.9:
            # 모델 등록 → Staging 승격 → FastAPI 핫스왑
            result = mlflow.register_model(
                model_uri=f"runs:/{run.info.run_id}/model",
                name="best_model"
            )
            time.sleep(3)  # DB 업데이트 대기

            client = MlflowClient()
            client.transition_model_version_stage(
                name="best_model",
                version=result.version,
                stage="Staging",
                archive_existing_versions=False
            )

            # FastAPI 핫스왑 요청
            requests.post("http://fastapi.local/reload")

        return acc

전체 코드: GitHub (train_and_log_model_fastapi.py)

FastAPI 코드는 5단계에서 구성한 것과 동일합니다. /reload 엔드포인트가 load_model_from_mlflow()를 호출하여 Staging 모델을 다시 로딩합니다.

검증 항목:

  • logreg_C, logreg_max_iter 파라미터로 학습 DAG 설정 가능
  • 모델 등록 시 Staging으로 자동 승격
  • FastAPI /reload 호출 시 핫스왑 적용 확인
  • /model-info, /predict에서 새 모델 정보 및 예측 결과 확인

설계 판단 (Why This Way?)#

  • BranchPythonOperator vs ShortCircuitOperator: BranchPythonOperator는 정확도에 따라 성공/실패 두 경로로 명시적으로 분기합니다. ShortCircuitOperator는 단순 skip 처리만 가능하므로, 실패 시 별도 알림이 필요한 경우 BranchPythonOperator가 적합합니다.
  • accuracy 임계값 하드코딩 vs Variable: 이 실험에서는 0.9를 코드에 직접 넣었지만, 프로덕션에서는 Airflow Variable이나 SSOT 정책 파일(policy.py)로 관리해야 합니다. 임계값 변경 시 코드 배포 없이 조정할 수 있습니다.
  • HTTP reload vs Pod restart: /reload API로 모델만 교체하면 Pod 재시작 없이 핫스왑이 가능합니다. Pod restart 방식은 안전하지만 다운타임이 발생합니다. 이 실험에서는 HTTP reload 방식을 선택하고, 프로덕션에서는 헬스체크와 롤백을 추가합니다.
  • time.sleep(3) 대기: MLflow 모델 등록 직후 DB 반영에 시간이 걸립니다. 프로덕션에서는 polling이나 이벤트 기반 대기로 대체해야 합니다.
  • 이 구조의 한계: 전역 변수 교체 방식의 핫스왑, HTTP 통신 실패 시 롤백 없음, 단일 Stage(Staging) 의존 등. Level 3에서 GitOps 기반 운영 고도화와 Triton 서빙으로 이 한계들을 개선합니다.

면접 포인트#

  • Q: 모델 품질 게이트(Quality Gate)란? → 학습된 모델이 서빙에 배포되기 전에 통과해야 하는 성능 기준입니다. 정확도, 지연시간, 데이터 드리프트 등을 자동으로 검증하여, 기준 미달 시 배포를 차단합니다. 이 실험에서는 accuracy > 0.9을 게이트로 사용했습니다.

  • Q: Airflow에서 조건부 분기를 구현하는 방법은? → BranchPythonOperator를 사용하여 Python 함수의 반환값으로 다음 실행할 task_id를 선택합니다. 반환되지 않은 경로의 Task는 자동으로 skip 처리됩니다.

  • Q: MLflow에서 모델 등록과 Stage 전환의 차이는? → register_model()은 모델을 Registry에 새 버전으로 등록합니다. transition_model_version_stage()는 등록된 버전의 Stage(None → Staging → Production)를 변경합니다. 등록과 승격은 별개 작업이므로 각각 호출해야 합니다.
