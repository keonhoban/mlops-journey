[Kubernetes - 3단계 : Ingress & Nginx Controller]

June 5, 2025 · 3 min

이 글에서 다루는 것#

여러 서비스를 하나의 진입점에서 경로(Path) 기반으로 라우팅하는 Ingress 리소스와 Nginx Ingress Controller를 다룹니다.

선수지식#

  • Kubernetes 1단계: Deployment & Service — Pod 배포와 Service 노출
  • Kubernetes 2단계: ConfigMap & Secret — 설정 분리 개념

이 단계에서 해결하려는 문제#

Service마다 NodePort를 열면 포트 관리가 복잡해지고, 도메인 기반 접근이 불가능합니다. Ingress는 단일 진입점에서 HTTP 요청을 경로나 호스트 기반으로 내부 서비스에 라우팅합니다. 실무에서 MLflow, Jupyter, API 서버를 하나의 도메인 아래 경로로 분리할 때 필수적인 구조입니다.

실습 코드: GitHub (Ingress_and_Nginx_Controller)

🧭 전체 흐름 요약#

① Ingress Controller 설치 (minikube addons 사용)
② nginx / httpd 서비스 배포
③ Ingress 리소스로 경로 라우팅 설정
④ /etc/hosts 수정 → curl 테스트로 확인

📦 실습 디렉토리 구조#

k8s-ingress/
├── deploy-nginx.yaml      # nginx 배포 + 서비스
├── deploy-httpd.yaml      # httpd 배포 + 서비스
├── ingress.yaml           # Ingress 라우팅 정의
└── README.md

⚙️ [1단계] Ingress Controller 설치#

minikube addons enable ingress
kubectl get pods -n ingress-nginx
  • ingress-nginx-controller Pod가 Running 상태면 설치 성공
  • NodePort 기본 포트: 80, 443

🧱 [2단계] 서비스 2개 배포 (nginx / httpd)#

deploy-nginx.yaml (핵심 부분)#

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80

deploy-httpd.yaml도 동일 구조 (image: httpd, name: httpd-*). 전체 코드: GitHub (Ingress_and_Nginx_Controller)

🗺️ [3단계] Ingress 라우팅 설정#

ingress.yaml#

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: example-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: foo.local
    http:
      paths:
      - path: /nginx
        pathType: Prefix
        backend:
          service:
            name: nginx-service
            port:
              number: 80
      - path: /httpd
        pathType: Prefix
        backend:
          service:
            name: httpd-service
            port:
              number: 80

🚀 [4단계] 실행 및 테스트#

kubectl apply -f deploy-nginx.yaml
kubectl apply -f deploy-httpd.yaml
kubectl apply -f ingress.yaml

minikube ip
# 예: 192.168.49.2

sudo nano /etc/hosts
# 아래 줄 추가
192.168.49.2   foo.local

curl http://foo.local/nginx    # nginx 화면 반환
curl http://foo.local/httpd    # httpd 페이지 반환

🎯 정리 요약#

항목내용
IngressHTTP(S) 요청을 Path/Host 기반으로 내부 서비스에 라우팅
Ingress Controller실제 라우팅 처리 담당 (Nginx 등)
/etc/hosts로컬 DNS 역할 (클러스터 외부 접근 가능하게 설정)
활용 사례/api, /mlflow, /jupyter 등 URL별 서비스 분리 가능

설계 판단 (Why This Way?)#

  • Ingress vs 서비스별 NodePort: NodePort는 포트 충돌과 관리 복잡도가 높습니다. Ingress는 단일 포트(80/443)에서 경로 기반으로 라우팅하여 관리가 간편합니다.
  • Nginx Ingress Controller 선택: 가장 널리 사용되고 문서가 풍부합니다. 대안으로 Traefik, Kong, Ambassador 등이 있지만, MLOps 스택에서는 Nginx가 표준적입니다.
  • rewrite-target 어노테이션: /nginx 경로 요청이 백엔드에는 /로 도달하도록 URL을 재작성합니다. 이것 없이는 백엔드에서 /nginx 경로를 처리해야 합니다.
  • /etc/hosts는 개발 환경 전용: 프로덕션에서는 DNS 서버와 연동하고, TLS 인증서(cert-manager)를 적용합니다.

면접 포인트#

  • Q: Ingress와 Service의 차이는? → Service는 L4(TCP/UDP) 레벨의 내부 로드밸런싱, Ingress는 L7(HTTP) 레벨의 호스트/경로 기반 라우팅입니다.

  • Q: Ingress Controller 없이 Ingress 리소스만 만들면? → Ingress는 선언적 규칙일 뿐이고, 실제 라우팅을 처리하는 Controller가 없으면 아무 동작도 하지 않습니다.

  • Q: 실무에서 Ingress에 TLS를 적용하는 방법은? → cert-manager로 Let’s Encrypt 인증서를 자동 발급하고, Ingress의 tls 섹션에 Secret을 연결합니다.
