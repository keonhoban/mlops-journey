[Kubernetes - 2단계: ConfigMap & Secret]

June 5, 2025 · 3 min

이 글에서 다루는 것#

애플리케이션 설정값(ConfigMap)과 민감 정보(Secret)를 코드에서 분리하여 Kubernetes 리소스로 관리하는 방법을 다룹니다.

선수지식#

  • Kubernetes 1단계: Deployment & Service — Pod 배포와 Service 기본 개념

이 단계에서 해결하려는 문제#

애플리케이션 코드에 DB 비밀번호나 환경 설정을 하드코딩하면, 환경별 배포가 어렵고 보안 위험이 생깁니다. Kubernetes의 ConfigMap과 Secret을 사용하면 설정과 코드를 분리하고, 환경 변수로 Pod에 주입할 수 있습니다.

실습 코드: GitHub (Configmap_and_Secret)

🧭 전체 흐름 요약#

① ConfigMap 생성 (일반 설정값)
② Secret 생성 (민감 정보 – base64 인코딩)
③ Pod에 환경 변수로 주입
④ 컨테이너 안에서 값 확인 (env 명령어)
⑤ 실전 적용 및 보안 주의사항 학습

📂 실습에 사용된 파일 구성#

k8s-configmap-secret/
├── configmap.yaml     # 일반 설정값 정의
├── secret.yaml        # 민감 정보 (base64 인코딩)
├── pod-env.yaml       # Pod 환경 변수로 주입
└── README.md

📄 주요 YAML 파일 정리#

configmap.yaml#

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_MODE: "production"
  APP_PORT: "8080"

secret.yaml#

apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  DB_PASSWORD: c3VwZXJzZWNyZXQ=   # base64 → "supersecret"

인코딩 방법: echo -n "supersecret" | base64

pod-env.yaml#

apiVersion: v1
kind: Pod
metadata:
  name: env-demo
spec:
  containers:
  - name: demo-container
    image: busybox
    command: [ "sh", "-c", "env; sleep 3600" ]
    env:
    - name: APP_MODE
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: APP_MODE
    - name: APP_PORT
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: APP_PORT
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: app-secret
          key: DB_PASSWORD

🧪 실습 실행 및 검증#

kubectl apply -f configmap.yaml
kubectl apply -f secret.yaml
kubectl apply -f pod-env.yaml

kubectl get pods
kubectl describe pod env-demo
kubectl exec -it env-demo -- sh

# 컨테이너 내부에서 확인
env

# 출력 예시 (중간 생략)
APP_MODE=production
APP_PORT=8080
DB_PASSWORD=supersecret

🎯 정리 요약#

구분설명
ConfigMap비민감 설정값 저장 (APP_MODE, APP_PORT 등)
Secret민감한 데이터 저장 (비밀번호, API 키 등) – base64 인코딩
주입 방식env를 통해 컨테이너에 주입
보안 주의Secret은 인코딩일 뿐, RBAC 및 접근 제어 필수

설계 판단 (Why This Way?)#

  • 환경 변수 vs Volume 마운트: 간단한 키-값은 환경 변수 주입이 편리합니다. 설정 파일 전체를 넘겨야 하면 Volume 마운트 방식을 사용합니다.
  • Secret의 base64는 암호화가 아님: etcd에 평문으로 저장되므로, 프로덕션에서는 SealedSecrets이나 외부 시크릿 관리(Vault, AWS Secrets Manager)를 사용해야 합니다.
  • ConfigMap 변경 시 Pod 반영: 환경 변수 방식은 Pod 재시작이 필요합니다. Volume 마운트 방식은 일정 시간 후 자동 반영됩니다.

면접 포인트#

  • Q: ConfigMap과 Secret의 차이는? → ConfigMap은 일반 설정값, Secret은 민감 정보용입니다. Secret은 base64 인코딩되며 RBAC으로 접근을 제한할 수 있습니다.

  • Q: Secret이 실제로 안전한가? → base64는 인코딩일 뿐 암호화가 아닙니다. etcd 암호화, RBAC 설정, SealedSecrets 등 추가 보안 조치가 필요합니다.

  • Q: ConfigMap 변경 후 Pod에 어떻게 반영하나? → 환경 변수 방식은 Pod 재시작 필요, Volume 마운트 방식은 kubelet이 주기적으로 갱신합니다.
